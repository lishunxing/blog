# 简介

此文档解释了JAVA并发编程的基础概念， 对于JAVA中线程的创建、生命周期进行了分析。讲解了线程阻塞、等待的操作步骤以及如何优雅的关闭掉一个线程。

对于并发带来的问题以及解决办法进行了讲述，解释了锁的升级过程以及对 synchronized、volatile 关键字进行了详解。

# 什么是并发、并行、串行

**并发**: 并发指的是一个系统能够在时间段内同时处理多个任务, 多线程程序在一个核的cpu上运行，就是并发. 比如多个人同时从一个门上地铁

**并行**: 并行是指系统同时执行多个任务, 多线程程序在多个核的cpu上运行，就是并行. 比如多个人同时从多个门上地铁

**串行**: 是指多个任务时，各个任务按照顺序执行，完成一个之后才能进行下一个. 比如多个人排队从一个门上地铁

**并发与并行的区别**：并发强调的是在单位时间内有多个任务同时进行，但是这些任务可能是交替执行的。而并行强调的是在同一时刻有多个任务同时进行。

一个系统可以同时处理多个任务，但这些任务可能只是交替执行，这是并发；而如果多个任务确实同时执行，那么这是并行。

> windows给每个cpu分配的时间片默认是20毫秒  如果当前线程执行不完20毫秒就会被释放  其他线程也最多会执行20毫秒的时间就会被切换到下一个线程去执行

## 线程状态

1.  NEW (新建)

    *   状态：线程对象被 `new` 创建出来，但尚未调用 `start()` 方法。
    *   特征：此时只是一个空的对象，还没有分配系统资源，也没有被操作系统调度。
2.  RUNNABLE(可运行)

    *   状态：调用了 `start()` 方法后进入该状态。
    *   特征：严格来说，在操作系统层面，这对应了 就绪（Ready） 和 运行（Running） 两种子状态。

        *   就绪：线程在等待 CPU 时间片。
        *   运行：线程正在被 CPU 执行。
        *   *之所以合二为一，是因为 Java 将获取 CPU 执行权的过程交给操作系统管理，JVM 不区分这两者。*
3.  WATTING (等待)

    *   状态：无限期等待，直到被其他线程显式唤醒。
    *   进入方式：

        *   `Object.wait()` （不设置超时时间）
        *   `Thread.join()` （不设置超时时间）
        *   `LockSupport.park()`
    *   特征：该状态没有超时时间，必须等待另一个线程执行 `notify()` / `notifyAll()` 或 `unpark()` 才能恢复。
4.  TIME\_WATTING (限期等待)

    *   状态：有明确等待时间的等待状态。
    *   进入方式：

        *   `Thread.sleep(long millis)`
        *   `Object.wait(long timeout)`
        *   `Thread.join(long millis)`
        *   `LockSupport.parkNanos() 指定等待时间`
        *   `LockSupport#parkUntil() 指定一个具体的唤醒时间`
    *   特征：时间到达后，系统会自动唤醒线程，返回到 RUNNABLE 状态。
5.  BLOCKED (阻塞)

    *   状态：线程在等待进入 同步代码块/方法 时的锁资源。
    *   特征：当线程试图获取一个被其他线程持有的对象锁（`synchronized`）时，会从 RUNNABLE 状态转入 BLOCKED 状态。这是一个 被动 的等待状态。
6.  TERMINATED (终止)

    *   状态：线程执行完毕（`run()` 方法结束）或因未捕获的异常而终止。
    *   特征：线程生命周期结束，不可再次 `start()`。

### sleep() - 线程休眠

```java
// 基本用法
Thread.sleep(1000);  // 休眠 1 秒

synchronized (lock) {
	Thread.sleep(5000);  // 持有 lock 休眠 5 秒
	// 其他线程无法获取 lock
}
```

核心特点：

*   不释放锁：持有锁时调用 `sleep()`，其他线程无法获取锁
*   自动唤醒：时间到达后自动进入就绪状态
*   属于 Thread 类：作用于当前线程
*   中断响应：抛出 `InterruptedException`

### wait() - 对象等待

```java
synchronized (obj) {
	obj.wait();           // 无限等待
    obj.wait(1000);       // 等待 1 秒
    obj.wait(1000, 500);  // 等待 1 秒 0.5 毫秒
	obj.wait();  // 释放 lock 锁，进入等待
	// 被唤醒后重新竞争 lock 锁
}
```

核心特点：

*   必须持有锁：否则抛出 `IllegalMonitorStateException`
*   释放锁：进入等待状态时释放对象锁
*   需要显式唤醒：必须由其他线程调用 `notify()`/`notifyAll()`
*   属于 Object 类：任何对象都可作为锁

### join() - 线程等待

```java
// 基本用法
Thread t = new Thread(() -> {});
t.start();
t.join();         // 等待 t 结束
t.join(1000);     // 最多等待 1 秒
```

核心特点：

*   等待线程死亡：当前线程等待目标线程执行完毕
*   底层调用 wait()：源码中调用了 `wait()`
*   不释放锁：不会释放任何业务锁
*   属于 Thread 类：作用于线程实例

### park() - 线程阻塞

```java
// 基本用法
LockSupport.park();                    // 无限阻塞
LockSupport.parkNanos(1000);           // 阻塞 1 微秒
LockSupport.parkUntil(System.currentTimeMillis() + 5000);  // 阻塞到绝对时间
LockSupport.unpark(thread);            // 唤醒指定线程
// 先 unpark 后 park 也能生效
LockSupport.unpark(Thread.currentThread());
LockSupport.park();  // 立即返回，不会阻塞
```

核心特点：

*   无需锁：可以直接调用
*   许可机制：每个线程有一个 0/1 的许可证
*   先唤醒后等待有效：如果先 `unpark()` 后 `park()`，`park()` 立即返回
*   精确唤醒：可以唤醒指定线程
*   中断不抛异常：只返回，需自行检查中断状态

总结:

*   sleep：休眠不释放锁
*   wait：等待释放锁
*   join：等待线程死
*   park：灵活阻塞原语

这四种方式各有侧重，`sleep/wait/join` 是面向业务的高级 API，而 `park` 是底层的构建基石，为 `ReentrantLock`、`CountDownLatch` 等高级并发工具提供了基础能力。

## 线程终止

### 正常终止（自然死亡）

线程的 run() 方法正常执行完毕，自然结束。

```java
Thread t = new Thread(() -> {
    System.out.println("执行任务");
    // 方法执行完毕，线程自然终止
});
t.start();
// t 执行完 run() 后自动进入 TERMINATED 状态
```

特点：

*   最安全、最优雅的终止方式
*   资源会被 JVM 自动回收
*   不会产生数据不一致问题

### 异常终止（意外死亡）

线程在运行过程中抛出未捕获的异常或错误，导致线程提前结束。

```java
Thread t = new Thread(() -> {
    throw new RuntimeException("未捕获的异常");  // 线程终止
});
t.start();
// 线程因异常而终止
```

### 强制终止（被迫死亡）

通过外部干预强制终止线程，但不推荐使用。

```java
// ❌ 已废弃的方法（Java 1.0）
thread.stop();  // 强制终止，可能导致数据不一致

// ⚠️ 不推荐的方法
thread.suspend();   // 挂起线程（已废弃）
thread.resume();    // 恢复线程（已废弃）
thread.destroy();   // 从未实现过 (已废弃)
```

为什么不推荐 `stop()`？

*   立即释放所有锁，可能导致对象状态不一致
*   可能造成数据损坏
*   无法保证资源正确清理

### 优雅终止线程的方式

使用 `interrupt()` 方法来发送中断信号给指定线程, `interrupt()` 不是"停止线程"，而是给线程发送一个"请中断"的信号。线程是否响应这个信号，完全由线程自己决定。

```java
// interrupt() 的真相
thread.interrupt();  // 不是停止线程，只是设置一个标志位 每个线程都有一个内部的 中断标志位（interrupt flag），interrupt() 会将其设置为 true。
```

*   `interrupt()` 就像给你的同事发消息："请停止你手上的工作"
*   但同事可以选择：

    *   立即停止
    *   完成当前步骤后停止
    *   完全忽略你的消息

对于非阻塞的方法, 可以通过判断线程的 `isInterrupted()` 状态来判断是否选择中断

```java
public class InterruptFlagDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            System.out.println("线程开始");
            
            // 检查中断标志位
            while (!Thread.currentThread().isInterrupted()) {
                // 正常工作
                System.out.println("工作中...");
            }
            
            System.out.println("收到中断信号，退出");
        });
        
        t.start();
        Thread.sleep(100);
        t.interrupt();  // 设置标志位为 true
    }
}
```

对于阻塞方法, 调用 interrupt() 则会抛出异常, 并且清除线程的 interrupt 标志位为 false

```java
public class InterruptSleepDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            try {
                System.out.println("开始睡眠 10 秒");
                Thread.sleep(10000);  // 会被中断
                System.out.println("睡眠结束");  // 不会执行
            } catch (InterruptedException e) {
                System.out.println("我被中断了！");
                // 注意：此时中断标志位已被清除
                System.out.println("中断标志位: " + Thread.currentThread().isInterrupted());  // false
            }
        });
        
        t.start();
        Thread.sleep(1000);  // 等 1 秒
        t.interrupt();       // 中断线程
    }
}
```

为什么 JVM 要清除标志位

1.  避免重复处理：一次中断应该只触发一次 `InterruptedException`
2.  清晰的语义：异常已经处理，不应该再保留标志, 避免在异常处理完后，中断标志还残留，导致后续代码误判
3.  兼容性：保持与 `wait()`、`join()` 等传统方法的一致性

为什么建议在 InterruptedException 中要恢复标志位

```java
// ❌ 错误示例：中断信号丢失
public class LostInterrupt {
    
    static void level1() {
        try {
            level2();
        } catch (InterruptedException e) {
            // 什么都不做，中断信号丢失！
            System.out.println("level1 捕获异常但不恢复");
        }
    }
    
    static void level2() throws InterruptedException {
        Thread.sleep(5000);  // 可能抛出 InterruptedException
    }
    
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            level1();
            // 这里检查中断状态
            if (Thread.currentThread().isInterrupted()) {
                System.out.println("检测到中断");  // 永远不会执行！
            } else {
                System.out.println("没有中断信号");  // 输出这个
            }
        });
        
        t.start();
        Thread.sleep(100);
        t.interrupt();  // 发送中断
        t.join();
    }
}
```

中断信号在 level1 的 catch 块中被"吞掉"了，上层完全不知道线程被中断过。

中断是协作机制，每个层级都应该尊重并传递中断信号, 需要让上层代码知道被中断了, 中断信号丢失，线程无法正确响应停止请求.&#x20;

所以要么恢复标志，要么重新抛出异常

# 并发带来的问题

并发编程的目的是为了提高程序的执行效率，但同时也引入了一系列复杂的问题，可能出现原子性问题，可见性问题以及有序性问题。

## 原子性问题

一个或多个操作在 CPU 执行时被线程调度器中断，导致操作未完整执行。

经典场景：`count++`   并非原子操作（包含读取-修改-写入三步）。两个线程同时执行  `count++` ，最终结果可能只加了一次（丢失更新）。

```java
public class AtomicityDemo {
    private static int count = 0;
    private static final int THREAD_COUNT = 10;
    private static final int INCREMENTS_PER_THREAD = 10000;
    
    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[THREAD_COUNT];
        
        for (int i = 0; i < THREAD_COUNT; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < INCREMENTS_PER_THREAD; j++) {
                    count++;  // 这不是原子操作！
                }
            });
            threads[i].start();
        }
        
        for (Thread t : threads) {
            t.join();
        }
        
        // 期望值：10 * 10000 = 100000
        System.out.println("实际结果: " + count);  // 实际输出了 23561 通常小于100000
    }
}
```

## 可见性问题;

一个线程对共享变量的修改，其他线程不能立即看到，导致读取到过期数据。

经典场景：

线程A修改了 flag = true，但修改只停留在 CPU 缓存中，未写回主内存。

线程B读取 flag 时，从自己的缓存或主内存中读到的仍是 false，导致无法正确响应状态变化。以下代码中, 线程reader永远无法读取到 writer 对于 flag的修改, 所以程序会一直死循环

```java
public class VisibilityDemo {
    private static boolean flag = true;  // 没有 volatile
    private static int value = 0;
    
    public static void main(String[] args) throws InterruptedException {
        Thread reader = new Thread(() -> {
            System.out.println("读取线程启动，等待 flag 变为 false...");
            int localValue = 0;
            
            // 死循环等待 flag 变为 false
            while (flag) {
                // 空循环等待
                localValue = value;
            }
            
            System.out.println("读取线程退出，value = " + value);
        });
        
        Thread writer = new Thread(() -> {
            try {
                Thread.sleep(1000);  // 等待1秒
                System.out.println("写入线程修改 flag = false");
                value = 100;
                flag = false;  // 修改共享变量
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });
        
        reader.start();
        writer.start();
        
        reader.join();  // 可能永远等不到，因为 reader 线程看不到 flag 的变化
    }
}
```

## 有序性问题

编译器或 CPU 为了优化性能，对指令进行重排序，在多线程环境下改变了程序的语义。

经典场景：双重检查锁（DCL）单例模式中，若无 volatile 修饰，对象初始化（分配内存、初始化、引用赋值）可能发生重排序，导致其他线程获取到未完全初始化的对象。

```java
public class Singleton {
    private static Singleton instance;  // 没有 volatile
    
    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();  // 问题在这里！
                }
            }
        }
        return instance;
    }
    
    private Singleton() {
        // 模拟复杂的初始化操作
        System.out.println("创建实例");
    }
}
```

这个代码看起来没问题：

*   第一次检查避免每次都加锁（性能优化）
*   第二次检查确保真的没创建才创建（线程安全）
*   很多人会认为这就是完美的单例模式

**但实际上，在某些情况下，它会返回一个未初始化完成的对象**

问题的根源：`new Singleton()` 不是原子操作, new Singleton() 在底层到底做了什么？创建一个对象，CPU 需要执行三个步骤：

```java
// 伪代码：new Singleton() 的三步骤
memory = allocate();           // 步骤1：分配内存空间
ctor(memory);                  // 步骤2：调用构造函数初始化对象
instance = memory;             // 步骤3：将引用指向内存地址
```

重排序是怎么发生的?

正常情况下的执行顺序应该是这样

线程A执行： 步骤1：分配内存 步骤2：初始化对象 步骤3：instance指向内存

此时对象完全初始化好了 ✅

但是CPU重排序后可能是这样

    时间线：
    T1: 线程A执行 instance = new Singleton();
        - 分配内存
        - instance 指向内存（此时对象未初始化）
        - 准备执行初始化...（被中断）
        
    T2: 线程B执行 getInstance()
        - if (instance == null)  → false（instance 已指向内存）
        - 直接返回 instance
        - 使用未初始化的对象 → 程序崩溃！

所以如果线程A和B同时调用方法获取 instance 假设线程A此时只执行了步骤1和3  准备执行步骤2（初始化）但还没执行完, 可能是因为有耗时的IO操作引起. 但是线程B此时进来拿到了一个并未真正完成初始化的instance, 造成空指针异常

# MESI 协议

MESI 协议（Modified-Exclusive-Shared-Invalid）是一种缓存一致性协议，用于保证多核 CPU 中各个核心的缓存数据一致性。它是现代 CPU 实现缓存一致性的基础。

```text
现代多核 CPU 架构：
┌─────────────────────────────────────────────────────────┐
│                    CPU Package                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │ Core 0   │ │ Core 1   │ │ Core 2   │ │ Core 3   │  │
│  │ L1 Cache │ │ L1 Cache │ │ L1 Cache │ │ L1 Cache │  │
│  │ L2 Cache │ │ L2 Cache │ │ L2 Cache │ │ L2 Cache │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
│       └────────────┴────────────┴────────────┘         │
│                    ┌─────────────────┐                  │
│                    │   L3 Cache      │                  │
│                    │   (共享)        │                  │
│                    └────────┬────────┘                  │
└─────────────────────────────┼──────────────────────────┘
                              ▼
                    ┌─────────────────┐
                    │   主内存        │
                    └─────────────────┘
// 初始状态：变量 x = 0
// Core 0 执行：x = 1
// Core 1 执行：int y = x

问题：
Core 0 将 x=1 写入自己的缓存
Core 1 的缓存中 x 还是 0
Core 1 读取到的是 0，而不是 1

需要 MESI 协议来解决！
```

每个缓存行（Cache Line，通常 64 字节）有四种状态：

| 状态            | 缩写 | 含义  | 说明                          |
| :------------ | :- | :-- | :-------------------------- |
| **Modified**  | M  | 已修改 | 当前核心已修改，数据与主内存不一致，且只有本核心有副本 |
| **Exclusive** | E  | 独占  | 当前核心有副本，与其他核心一致，且只有本核心有副本   |
| **Shared**    | S  | 共享  | 当前核心有副本，与其他核心一致，可能有多个核心有副本  |
| **Invalid**   | I  | 无效  | 当前核心的副本已失效，不能使用             |

正常的MESI工作流程

```code
Core 0 执行 x = 1：
写入操作太慢，需要等待其他核心的 Invalidate Acknowledge

1. 发送 Invalidate 消息
2. 等待 Core 1, Core 2, ... 的确认 ← 慢！
3. 才能写入缓存
```

优化方法, 引入了 Store Buffer（存储缓冲区）

```code
Core 0 执行 x = 1：

1. 将写入放入 Store Buffer（立即返回）
2. 发送 Invalidate 消息
3. 继续执行其他指令
4. 收到所有确认后，从 Store Buffer 写入缓存

问题：其他核心看不到 Store Buffer 中的数据！
```

同时对于其他线程的数据修改, 引入了 Invalidate Queue（失效队列）, 但是又有了新的问题, 处理 Invalidate 消息也需要时间

```code
Core 1 收到 Invalidate 消息：

1. 放入 Invalidate Queue（立即响应）
2. 稍后真正失效缓存

问题：Core 1 可能还在使用已失效的数据！
```

所以在多线程的环境下, 因为CPU各个内核的同步的原因, 并发操作会有问题. 这些问题在JAVA中可以通过 `synchronize` 和 `volatile` 来解决

# synchronize 关键字

使用synchronize关键字可以对类或者对象进行加锁, 来让并发操作变成串行操作. 保证同一时刻最多只有一个线程执行被修饰的代码，从而保证线程安全。

他可以解决并发带来的原子性，有序性和可见性问题

## 原子性

确保被保护的代码块在执行过程中不会被其他线程干扰，表现为"要么全部执行成功，要么都不执行"。

*   对于非静态方法：锁定的是调用该方法的实例对象（`this`）。同一时刻，对于同一个实例，只有一个线程能进入该方法。
*   对于静态方法：锁定的是该类的Class对象。因为类对象在JVM中只有一个，所以它能保证所有实例的同步，同一时刻只有一个线程能执行该静态方法。
*   对于代码块：可以更灵活地指定锁对象，控制锁的粒度（例如 `synchronized (lock) {}`）。

## 可见性

Java 内存模型（JMM）中，每个线程有自己的工作内存（本地缓存）。synchronized 在加锁和解锁时会强制执行缓存与主内存的同步：

进入同步块前：清空工作内存，从主内存中重新读取共享变量。

退出同步块时：将工作内存中修改的变量值刷新回主内存。

这保证了一个线程修改的共享变量，能立即被其他线程（在获取同一把锁后）看到，避免了"脏读"。

## 有序性

虽然编译器（JIT）和CPU为了性能可能会对指令进行重排序，但 synchronized 限制了并发执行的逻辑。 严格来说，它不禁止代码内部的重排序（单线程内重排序不影响结果），但它保证了持有同一把锁的多个线程之间，代码的执行是串行的。这种"串行"特性实际上起到了防止多线程环境下因重排序导致逻辑错误的作用。

## 底层原理实现

在 JVM 层面，`synchronized` 是通过 监视器锁 实现的：

*   当线程进入同步代码块时，它必须获得该对象的 监视器。
*   每个对象在内存中都有一个对象头（Mark Word），其中记录了锁的状态和持有锁的线程ID。
*   在 JDK 1.6 之后，JVM 对 `synchronized` 进行了大量优化（锁升级）：

    1.  无锁 -> 2. 偏向锁（只有一个线程） -> 3. 轻量级锁（少量竞争，通过CAS自旋） -> 4. 重量级锁（大量竞争，线程阻塞，依赖操作系统互斥量）。

    *   得益于这种优化，`synchronized` 在低并发场景下的性能已经非常接近 `ReentrantLock`，甚至在某些场景下更优（因为 JVM 可以自动优化）。
*   锁升级代码示例

```java
public class ObservableLockUpgrade {
    private static final Object lock = new Object();
    
    public static void main(String[] args) throws InterruptedException {
        System.out.println("=== 偏向锁演示 ===");
        biasedLockDemo();
        
        System.out.println("\n=== 轻量级锁演示 ===");
        lightLockDemo();
        
        System.out.println("\n=== 重量级锁演示 ===");
        heavyLockDemo();
    }
    
    // 偏向锁：单线程反复获取锁
    private static void biasedLockDemo() throws InterruptedException {
        Thread thread = new Thread(() -> {
            for (int i = 0; i < 10; i++) {
                synchronized (lock) {
                    System.out.println(Thread.currentThread().getName() + " 获取锁，第" + i + "次");
                    // 偏向锁不会主动释放，同一线程重入无开销
                }
                try {
                    Thread.sleep(10);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });
        
        thread.start();
        thread.join();
        
        // 注意：此时锁还是偏向线程thread
        System.out.println("偏向锁仍然偏向于：" + thread.getName());
    }
    
    // 轻量级锁：两个线程交替执行，无竞争
    private static void lightLockDemo() throws InterruptedException {
        // 先让偏向锁失效（因为上一个锁偏向于其他线程）
        Thread.sleep(100);
        
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                System.out.println("线程1 获取锁，执行耗时操作");
                try {
                    Thread.sleep(50);  // 持有锁一段时间
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });
        
        Thread t2 = new Thread(() -> {
            try {
                Thread.sleep(10);  // 确保t1先拿到锁
                synchronized (lock) {
                    System.out.println("线程2 获取锁");
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });
        
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        
        // 由于两个线程交替执行，没有同时竞争，会使用轻量级锁
        System.out.println("交替执行，使用了轻量级锁");
    }
    
    // 重量级锁：两个线程同时竞争
    private static void heavyLockDemo() throws InterruptedException {
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                System.out.println("线程1 获取锁，长时间持有");
                try {
                    Thread.sleep(1000);  // 长时间持有锁
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });
        
        Thread t2 = new Thread(() -> {
            System.out.println("线程2 等待锁...");
            synchronized (lock) {
                System.out.println("线程2 获取锁");
            }
        });
        
        t1.start();
        Thread.sleep(10);  // 确保t1先拿到锁
        t2.start();
        
        t1.join();
        t2.join();
        
        // t2 在等待 t1 释放锁时，会自旋一段时间
        // 自旋失败后，膨胀为重量级锁，线程阻塞
        System.out.println("同时竞争，升级为重量级锁");
    }
}
```

1.  锁升级是单向的：偏向锁 → 轻量级锁 → 重量级锁 不能降级，只能升级
2.  轻量级锁自旋没有固定的时间阈值，完全自适应. 锁持有时间 < 50 微秒通常能保持轻量级锁, 锁持有时间 > 1 毫秒几乎必然升级为重量级锁
3.  偏向锁在 JDK 15 默认关闭，JDK 18 已废弃
4.  锁升级是 JVM 自动完成的，无需程序员干预
5.  JDK 8 及以后版本，开发者无法精确控制自旋次数和时间，JVM 完全自适应

# volatile关键字

`volatile` 是 Java 中一个轻量级的同步关键字，它解决了并发编程中的可见性和有序性问题，但不保证原子性。

## 保证可见性

当一个线程修改了 volatile 变量，新值会立即写回主内存，其他线程读取时必须从主内存获取，而不是从 CPU 缓存读取。

## 禁止指令重排序

`volatile` 通过内存屏障，防止编译器和 CPU 对代码进行重排序，确保代码按编写顺序执行

## 重排序发生的层面

*   编译器重排序: JIT 编译器优化，重新安排指令顺序
*   CPU 乱序执行: CPU 动态调度，提前执行无依赖指令
*   内存系统重排序: Store Buffer、Invalidate Queue 导致写入/读取顺序变化

## CPU重排序的原因

指令重排序是 CPU 为了优化性能而做的一种"投机取巧"，核心目的是充分利用 CPU 资源，避免空闲等待。CPU 的执行速度远快于内存访问, 当 CPU 需要从内存读取数据时，可能要等待 200 纳秒，而这段时间 CPU 可以执行上百条指令

```code
速度对比（纳秒级）：
┌─────────────────────────────────────────────────────────┐
│ CPU 执行一条指令：    0.3 - 1 纳秒                      │
│ 从 L1 缓存读取：      1 - 2 纳秒                        │
│ 从 L2 缓存读取：      10 - 20 纳秒                      │
│ 从 L3 缓存读取：      30 - 50 纳秒                      │
│ 从主内存读取：        100 - 200 纳秒  ← 瓶颈！          │
└─────────────────────────────────────────────────────────┘
```

所以CPU 不会傻等，而是会先执行后面不依赖当前数据的指令。

## 内存重排序的原因

**Store Buffer（存储缓冲区**）CPU 写入内存时，不会直接写，而是先放到 Store Buffer, 在还没同步到主内存前旧值还是可能被读取到

```code
CPU 核心                  Store Buffer              主内存
┌─────────┐            ┌──────────────┐          ┌─────────┐
│ Core 0  │──写入──→  │ [A=1] [B=2]  │──异步──→ │ A=0     │
│         │            │ [C=3]        │          │ B=0     │
└─────────┘            └──────────────┘          │ C=0     │
                                                  └─────────┘
// 初始：A=0, B=0

// Core 0 执行：
A = 1;      // 写入 Store Buffer，还没到主内存
B = 1;      // 写入 Store Buffer，还没到主内存

// Core 1 执行：
if (B == 1) {
    // 可能看到 B=1，但 A 还是 0！
    assert A == 1;  // 可能失败！
}
```

**Invalidate Queue（失效队列）**

CPU 收到"缓存失效"通知时，不会立即处理，而是放入队列：

```code
Core 0 修改 A=1
    ↓
发送"缓存失效"消息
    ↓
Core 1 收到消息，放入 Invalidate Queue
    ↓
Core 1 继续执行（还没处理队列）
    ↓
Core 1 读取 A → 可能读到旧值！
```

## 内存屏障

内存屏障（Memory Barrier）是一种 CPU 或编译器指令，用于禁止指令重排序和保证内存可见性。

四种屏障的命名格式：操作1 + 操作2  Load：读操作（从内存加载数据）Store：写操作（存储数据到内存）

*   `LoadLoad`：第一个读操作 与 第二个读操作 之间的屏障, 屏障前的所有读操作必须在屏障后的读操作之前完成。
*   `StoreStore`：第一个写操作 与 第二个写操作 之间的屏障, 屏障前的所有写操作必须在屏障后的写操作之前完成。
*   `LoadStore`：读操作 与 写操作 之间的屏障, 屏障前的所有读操作必须在屏障后的写操作之前完成。
*   `StoreLoad`：写操作 与 读操作 之间的屏障（最严格）, 屏障前的所有写操作必须在屏障后的读操作之前完成。这是开销最大的屏障，因为它需要刷新 Store Buffer 和 Invalidate Queue。

x86 有较强的内存顺序保证（TSO - Total Store Order），只对写-读(StoreLoad) 重排序敏感， ARM 架构（需要所有屏障）

# Happen-Before 规则

Happen-Before 是 Java 内存模型（JMM）的核心概念，用于判断多线程环境下两个操作是否存在顺序关系，从而保证内存可见性。如果两个操作满足 happen-before 关系，那么：

*   顺序保证：第一个操作的结果对第二个操作可见
*   顺序保证：第一个操作的执行顺序在第二个操作之前

简单来说就是前一个操作的结果可以被后续的操作获取

## 程序顺序规则

在一个线程内，按照代码顺序，前面的操作 happen-before 后面的操作。

```java
public void programOrderRule() {
    int a = 1;        // A：写操作
    int b = a + 1;    // B：读操作
    // A happen-before B（保证 a 的值对 b 可见）
}
```

## 监视器锁规则

线程1的解锁（A）happen-before 线程2的加锁（B）, 因此，线程2读取 count 时，一定能看到线程1写入的值 100

```java
public class MonitorLockRule {
    private static int count = 0;
    private static final Object lock = new Object();
    
    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lock) {           // 加锁
                count = 100;                // 写操作
            }                               // 解锁（A）
        });
        
        Thread t2 = new Thread(() -> {
            synchronized (lock) {           // 加锁（B）
                int value = count;          // 读操作
                System.out.println(value);  // 保证看到 100
            }
        });
        
        t1.start();
        t2.start();
    }
}
```

## volatile 变量规则

对一个 volatile 变量的写操作 happen-before 后续对这个变量的读操作, 因为本身通过内存屏障来实现了。volatile 的 happen-before 具有传递性

```java
public class VolatileRule {
    private static volatile boolean flag = false;
    private static int value = 0;
    
    public static void main(String[] args) {
        Thread writer = new Thread(() -> {
            value = 100;           // 普通写
            flag = true;           // volatile 写（A）
        });
        
        Thread reader = new Thread(() -> {
            if (flag) {            // volatile 读（B）
                int v = value;     // 普通读
                System.out.println(v);  // 保证看到 100
            }
        });
        
        writer.start();
        reader.start();
    }
}
```

## 传递性规则

如果 A happen-before B，且 B happen-before C，那么 A happen-before C。

```java
public class TransitivityRule {
    private static int x = 0;
    private static volatile int y = 0;
    private static int z = 0;
    
    public static void main(String[] args) {
        // 线程1
        x = 1;           // A
        y = 2;           // B（volatile 写）
        
        // 线程2
        int temp = y;    // C（volatile 读）
        z = temp;        // D
        
        // 传递关系：
        // A happen-before B (程序顺序规则)
        // B happen-before C (volatile 规则)
        // C happen-before D (程序顺序规则)
        // 因此：A happen-before D
        // 线程2 的 z 一定能看到 x=1
    }
}
```

## 线程启动规则

调用 `Thread.start()` 方法 happen-before 新线程中的任何操作。

```java
public class StartRule {
    private static int value = 0;
    
    public static void main(String[] args) {
        Thread t = new Thread(() -> {
            // 这里的操作一定能看到主线程在 start() 之前的所有修改
            System.out.println(value);  // 保证看到 100
        });
        
        value = 100;           // A
        t.start();             // B：start() 方法
        // A happen-before B (程序顺序规则)
        // B happen-before 线程内操作 (线程启动规则)
        // 因此：value=100 对线程内可见
    }
}
```

## 线程终止规则

线程中的任何操作 happen-before 其他线程检测到该线程终止。

```java
public class JoinRule {
    private static int result = 0;
    
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            result = 100;       // 线程内的操作（A）
            // 线程结束
        });
        
        t.start();
        t.join();               // B：等待线程结束
        
        // A happen-before B (线程终止规则)
        System.out.println(result);  // 保证看到 100
    }
}
```

## 线程中断规则

调用 `Thread.interrupt()` 方法 happen-before 被中断线程检测到中断。

```java
public class InterruptRule {
    private static int value = 0;
    
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                // 忙等待
            }
            // 检测到中断后，一定能看到中断前的修改
            System.out.println(value);  // 保证看到 200
        });
        
        t.start();
        
        value = 200;            // A：修改共享变量
        t.interrupt();          // B：中断线程
        
        // A happen-before B (程序顺序规则)
        // B happen-before 线程检测到中断 (中断规则)
        // 因此：value=200 对线程可见
    }
}
```

## 对象终结规则

一个对象的初始化的完成，也就是构造函数执行的结束一定 happens-before它的finalize()方法

```java
public class FinalizeRule {
    private int value;
    
    public FinalizeRule() {
        value = 100;           // 构造函数初始化（A）
        // 构造完成
    }
    
    @Override
    protected void finalize() throws Throwable {
        // A happen-before finalize()
        System.out.println(value);  // 保证看到 100
        super.finalize();
    }
}
```

## 各屏障的规则实现

| **程序顺序规则**      | 无（编译器保证）                                        | 不重排序有依赖的指令      |
| :-------------- | :---------------------------------------------- | :-------------- |
| **监视器锁规则**      | `monitorenter` 前：无 `monitorexit` 后：StoreLoad    | 确保解锁后的修改对其他线程可见 |
| **volatile 规则** | 写：StoreStore + StoreLoad 读：LoadLoad + LoadStore | 禁止重排序 + 保证可见性   |
| **线程启动规则**      | start() 方法：StoreLoad                            | 确保启动前的修改对线程可见   |
| **线程终止规则**      | join() 方法：LoadLoad                              | 确保线程内的修改对调用者可见  |
| **线程中断规则**      | interrupt() 方法：StoreLoad                        | 确保中断标志的可见性      |
| **对象终结规则**      | 构造方法结束：StoreStore                               | 确保对象完全初始化       |
