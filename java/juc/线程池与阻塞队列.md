# 线程池与阻塞队列

线程池是一种池化技术，在系统启动时预先创建一定数量的线程，将这些线程集中管理，当有任务需要执行时，从池中取出空闲线程来处理，任务完成后线程返回池中等待下次使用，而不是销毁。

## 核心目的

### 1. 降低资源消耗

*   问题：频繁创建和销毁线程需要分配内存、注册等操作，开销大
*   解决：复用已有线程，避免反复创建/销毁

### 2. 提高响应速度

*   问题：任务到达时才创建线程，有延迟
*   解决：线程已预先创建，任务到达即可立即执行

### 3. 提高线程可管理性

*   问题：无限制创建线程会导致系统资源耗尽（OOM、崩溃）
*   解决：统一分配、调优、监控，限制线程数量上限

### 4. 提供任务排队与拒绝机制

*   任务过多时进入队列等待
*   超出负载时触发拒绝策略，防止系统过载

## 使用方式

### 通过 Executors 工厂类创建（快捷方式）

Java 提供了 Executors 工具类，可快速创建常见类型的线程池, JDK8 中有 5种线程池

| 类型                          | 创建方法                             | 线程数   | 队列类型 | 适用场景                                                     | 优点                                        | 缺点                                                         |
| :---------------------------- | :----------------------------------- | :------- | :------- | :----------------------------------------------------------- | :------------------------------------------ | :----------------------------------------------------------- |
| FixedThreadPool               | `newFixedThreadPool(int)`            | 固定大小 | 无界队列 | Web 服务器处理请求（如 Tomcat 线程池）/ 批量处理固定数量的任务 / 负载相对平稳的后台任务 | 线程数可控，资源稳定 / 响应时间可预测       | 无界队列可能导致 OOM / 任务积压时延迟增加                    |
| CachedThreadPool              | `newCachedThreadPool()`              | 动态增长 | 同步队列 | 大量短生命周期的任务（如异步发送邮件）/ 突发流量、任务执行时间极短 / 负载波动大的场景 | 响应快，任务到达立即执行 / 空闲线程自动回收 | 线程数无上限，可能耗尽内存 /  高并发下可能创建数万个线程，导致内存溢出或系统崩溃 |
| SingleThreadExecutor          | `newSingleThreadExecutor()`          | 单线程   | 无界队列 | 需要严格顺序执行的任务（如日志写入）/ 单线程访问的共享资源（如文件写入）/ 任务间依赖关系强 | 保证任务顺序执行 / 无需手动处理线程同步     | 单点故障，效率低 / 任务积压可能导致延迟                      |
| ScheduledThreadPool           | `newScheduledThreadPool(int)`        | 固定大小 | 延迟队列 | 心跳检测、健康检查 / 定时数据同步、报表生成/ 延迟重试机制    | 支持定时、延迟、周期执行 / 多线程并发执行   | 任务执行时间过长会影响后续调度 / 相对复杂，需处理异常        |
| SingleThreadScheduledExecutor | `newSingleThreadScheduledExecutor()` | 单线程   | 延迟队列 | 单线程定时任务                                               | 保证定时任务不并发执行 / 无需处理并发同步   | 一个任务阻塞，全部延迟                                       |

### FixedThreadPool（固定大小线程池）

```java
ExecutorService pool = Executors.newFixedThreadPool(5);

// 底层实现
return new ThreadPoolExecutor(
    nThreads, nThreads,    // 核心=最大=固定值
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<Runnable>()  // 无界队列
);
```

### CachedThreadPool（缓存线程池）

```java
ExecutorService pool = Executors.newCachedThreadPool();

// 底层实现
return new ThreadPoolExecutor(
    0, Integer.MAX_VALUE,  // 核心=0，最大≈无限
    60L, TimeUnit.SECONDS,
    new SynchronousQueue<Runnable>()  // 直接提交队列
);
```

### SingleThreadExecutor（单线程池）

```java
ExecutorService pool = Executors.newSingleThreadExecutor();

// 底层实现
return new FinalizableDelegatedExecutorService(
    new ThreadPoolExecutor(1, 1, 0L, TimeUnit.MILLISECONDS, new LinkedBlockingQueue<Runnable>())
);
```

### ScheduledThreadPool（定时线程池）

```java
ScheduledExecutorService pool = Executors.newScheduledThreadPool(3);
// 底层实现
public static ScheduledExecutorService newScheduledThreadPool(int corePoolSize) {
    return new ScheduledThreadPoolExecutor(corePoolSize);
}

// 延迟执行
pool.schedule(() -> System.out.println("延迟3秒"), 3, TimeUnit.SECONDS);

// 固定频率执行（忽略任务执行时间）
pool.scheduleAtFixedRate(() -> {
    System.out.println("每2秒执行一次");
}, 0, 2, TimeUnit.SECONDS);

// 固定延迟执行（等待上次任务完成）
pool.scheduleWithFixedDelay(() -> {
    Thread.sleep(1000);
    System.out.println("上次完成后延迟1秒");
}, 0, 1, TimeUnit.SECONDS);

```

### SingleThreadScheduledExecutor（单线程定时池）

```java
// 示例：单线程定时清理缓存，避免并发冲突
ScheduledExecutorService pool = Executors.newSingleThreadScheduledExecutor();
pool.scheduleAtFixedRate(() -> {
    cache.clearExpired();  // 保证不并发执行
}, 0, 30, TimeUnit.MINUTES);
```

### 通过 ThreadPoolExecutor 自定义创建）

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    5,                                  // 核心线程数
    10,                                 // 最大线程数
    60L, 							// 时间
    TimeUnit.SECONDS,              // 单位, 空闲线程60秒回收
    new ArrayBlockingQueue<>(100),      // 有界队列，容量100
    Executors.defaultThreadFactory(),   // 默认线程工厂
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：调用者运行
);

// 提交任务
pool.execute(() -> {
    System.out.println("执行任务：" + Thread.currentThread().getName());
});

// 提交带返回值的任务
Future<String> future = pool.submit(() -> {
    Thread.sleep(1000);
    return "结果";
});
String result = future.get();  // 阻塞获取结果

// 关闭线程池
pool.shutdown();     // 优雅关闭，等待已有任务完成
// pool.shutdownNow();  // 立即关闭，尝试中断正在执行的任务
```

## 线程工厂 ThreadFactory

ThreadFactory 是一个创建线程的工厂接口，用于统一控制线程的创建过程，包括：

*   设置线程名称
*   设置是否为守护线程
*   设置线程优先级
*   设置未捕获异常处理器

### 常用线程工厂实现

#### Executors.defaultThreadFactory()（JDK 默认）

```java
ThreadFactory factory = Executors.defaultThreadFactory();
Thread thread = factory.newThread(() -> System.out.println("任务执行"));
thread.start();
```

#### Guava 的 ThreadFactoryBuilder

```java
<!-- Maven 依赖 -->
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>32.1.3-jre</version>
</dependency>


ThreadFactory factory = new ThreadFactoryBuilder()
    .setNameFormat("my-pool-%d")           // 线程名格式，%d 自动递增
    .setDaemon(true)                        // 守护线程
    .setPriority(Thread.MAX_PRIORITY)       // 优先级
    .setUncaughtExceptionHandler((t, e) -> {
        log.error("线程 {} 异常", t.getName(), e);
    })
    .build();

ExecutorService pool = Executors.newFixedThreadPool(10, factory);
```

#### Apache Commons Lang3 的 BasicThreadFactory

```java
<dependency>
    <groupId>org.apache.commons</groupId>
    <artifactId>commons-lang3</artifactId>
    <version>3.13.0</version>
</dependency>

import org.apache.commons.lang3.concurrent.BasicThreadFactory;

ThreadFactory factory = new BasicThreadFactory.Builder()
    .namingPattern("worker-%d")      // 线程名格式
    .daemon(true)                     // 守护线程
    .priority(Thread.NORM_PRIORITY)   // 优先级
    .uncaughtExceptionHandler((t, e) -> {
        System.err.println(t.getName() + " 异常: " + e);
    })
    .build();
```

### 实践建议

1.  一定要给线程命名：线上排查问题时，`http-nio-8080-exec-3` 比 `Thread-5` 有用得多
2.  设置未捕获异常处理器：防止异常静默丢失
3.  业务隔离：不同业务使用不同的 ThreadFactory，如 `order-pool-`、`user-pool-`、`report-pool-`
4.  生产环境优先用 Guava 的 ThreadFactoryBuilder：代码清晰，功能完整

### 守护线程 vs 非守护线程（用户线程）

| 对比维度         | 守护线程 (Daemon Thread) | 非守护线程 (User Thread)     |
| :--------------- | :----------------------- | :--------------------------- |
| **JVM 退出行为** | JVM 不会等待守护线程结束 | JVM 必须等待所有用户线程结束 |
| **主要用途**     | 后台服务、辅助任务       | 核心业务逻辑                 |
| **生命周期**     | 随 JVM 退出而强制终止    | 独立运行直到任务完成         |
| **finally 执行** | 不保证执行               | 正常执行                     |

#### JVM 退出行为（最重要）

```java
// 非守护线程示例
Thread userThread = new Thread(() -> {
    for (int i = 0; i < 10; i++) {
        System.out.println("用户线程执行: " + i);
        Thread.sleep(1000);
    }
});
userThread.setDaemon(false);  // 默认就是 false
userThread.start();

System.out.println("main 线程结束");
// main 结束后，JVM 会等待 userThread 执行完才退出

// 守护线程示例
Thread daemonThread = new Thread(() -> {
    int i = 0;
    while (true) {  // 无限循环
        System.out.println("守护线程执行: " + i++);
        Thread.sleep(500);
    }
});
daemonThread.setDaemon(true);  // 设置为守护线程
daemonThread.start();

System.out.println("main 线程结束");
// main 结束后，JVM 立即退出，守护线程被强制终止
// 输出可能只打印几条就被强制结束了
```

#### finally 块不保证执行

```java
Thread daemonThread = new Thread(() -> {
    try {
        while (true) {
            System.out.println("工作中...");
            Thread.sleep(1000);
        }
    } finally {
        // JVM 退出时不会执行这里！
        System.out.println("释放资源...");  // 永远不会打印
    }
});
daemonThread.setDaemon(true);
daemonThread.start();
```

#### 守护线程适合的场景

*   垃圾回收 (GC) 
*   心跳检测
*   日志监控
*   缓存清理
*   JVM 监控线程

### JVM退出判断逻辑

1.  如果存在任何非守护线程（用户线程）还在运行 → JVM 继续运行，等待用户线程结束

2.  如果所有用户线程都已经结束 → JVM 开始退出 → 所有守护线程被立即强制终止（无论是否执行完成）

3.  特殊情况：只剩守护线程时 → JVM 立即退出

### 线程工厂的优先级参数 Priority

线程优先级是传递给操作系统的调度提示（Hint），JVM 会通过底层 API 将其映射到操作系统的线程优先级。

```java
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setPriority(Thread.MAX_PRIORITY);  // 设置为 10
    return t;
};
```

线程池内的所有线程都是相同的低优先级，它们之间没有区别。优先级在线程池中的实际意义是与其他线程池进行对比, 区分 UI 线程和后台线程

```java
// UI 线程高优先级，保证界面响应
Thread uiThread = new Thread(() -> renderUI());
uiThread.setPriority(Thread.MAX_PRIORITY);

// 后台日志线程低优先级，不干扰 UI
Thread logThread = new Thread(() -> writeLog());
logThread.setPriority(Thread.MIN_PRIORITY);
```

不同操作系统的映射差异

| 操作系统    | 优先级数量 | Java 优先级映射行为                             |
| :---------- | :--------- | :---------------------------------------------- |
| Linux (CFS) | 140        | 映射到 nice 值，但 CFS 调度器对优先级敏感度较低 |
| Windows     | 32         | 映射效果较好，差异明显                          |
| Solaris     | 约 2^31    | 映射效果好                                      |

Linux 的 CFS（完全公平调度器）对优先级敏感度不如 Windows 高，这是很多人觉得"设置了也没用"的主要原因。

## 阻塞队列

阻塞队列是线程池的核心组件，用于存储等待执行的任务。当队列为空时，获取元素的线程阻塞；当队列满时，添加元素的线程阻塞。JAVA中目前有7种阻塞队列

| 阻塞队列                  | 数据结构   | 有界/无界        | 锁机制              | 特点                 |
| :------------------------ | :--------- | :--------------- | :------------------ | :------------------- |
| **ArrayBlockingQueue**    | 数组       | 有界             | 单锁（公平/非公平） | 最常用，性能稳定     |
| **LinkedBlockingQueue**   | 链表       | 可选（默认无界） | 双锁（读写分离）    | 吞吐量高             |
| **SynchronousQueue**      | 无存储     | 特殊（容量0）    | CAS+自旋            | 直接传递，不存储     |
| **PriorityBlockingQueue** | 二叉堆     | 无界             | 单锁                | 支持优先级排序       |
| **DelayQueue**            | 优先级队列 | 无界             | 单锁                | 延迟获取元素         |
| **LinkedTransferQueue**   | 链表       | 无界             | 复杂CAS             | 支持生产者等待消费者 |
| **LinkedBlockingDeque**   | 双向链表   | 可选（默认无界） | 单锁                | 支持双端操作         |

### 队列常用方法

| 操作类型 | 抛出异常    | 返回特殊值 | 阻塞等待 | 超时退出               |
| :------- | :---------- | :--------- | :------- | :--------------------- |
| **插入** | `add(e)`    | `offer(e)` | `put(e)` | `offer(e, time, unit)` |
| **移除** | `remove()`  | `poll()`   | `take()` | `poll(time, unit)`     |
| **检查** | `element()` | `peek()`   | ❌ 不支持 | ❌ 不支持               |

```java
// 创建队列
BlockingQueue<String> q = new ArrayBlockingQueue<>(10);

// 插入操作
q.add("e");                    // 队列满时抛出 IllegalStateException
q.offer("e");                  // 返回 false
q.put("e");                    // 阻塞
q.offer("e", 1, TimeUnit.SECONDS); // 超时退出

// 移除操作
String e = q.remove();         // 队列空时抛出 NoSuchElementException
String e = q.poll();           // 返回 null
String e = q.take();           // 阻塞
String e = q.poll(1, TimeUnit.SECONDS); // 超时返回 null

// 检查操作（不删除）
String e = q.element();        // 	队列空时抛出 NoSuchElementException
String e = q.peek();           // 返回 null
```

### ArrayBlockingQueue（有界数组队列）

底层：循环数组 + ReentrantLock

```java
// 创建
ArrayBlockingQueue queue = new ArrayBlockingQueue(100);  // 必须指定容量
ArrayBlockingQueue queue = new ArrayBlockingQueue(100, true);  // 公平锁模式 实际就是把 ReentrantLock 改为了公平锁
```

特点：

*   容量固定，内存可控
*   公平模式下性能稍低，但能防止线程饥饿

    *   线程饥饿是指某个线程（或一组线程）因为长期无法获得所需的CPU时间或资源，导致无法正常执行的情况。

### LinkedBlockingQueue（链表队列）

底层：单向链表 + 两把锁（putLock/takeLock）

```java
// 创建
LinkedBlockingQueue queue = new LinkedBlockingQueue();      // 无界（Integer.MAX_VALUE）
LinkedBlockingQueue queue = new LinkedBlockingQueue(100);   // 有界
```

特点：

*   读写分离，吞吐量高于 ArrayBlockingQueue
*   无界模式下可能导致内存溢出

### SynchronousQueue（同步队列）

```java
// 创建
SynchronousQueue queue = new SynchronousQueue();          // 非公平
SynchronousQueue queue = new SynchronousQueue(true);     // 公平模式（FIFO）

// CachedThreadPool 内部使用
SynchronousQueue<Runnable> queue = new SynchronousQueue<>();

// 必须成对出现
new Thread(() -> {
    try {
        queue.put("data");  // 阻塞等待 take
    } catch (InterruptedException e) {}
}).start();

new Thread(() -> {
    try {
        String data = queue.take();  // 阻塞等待 put
    } catch (InterruptedException e) {}
}).start();
```

特点：

*   每个插入操作必须等待对应的移除操作
*   适合直接传递任务，无缓冲

### PriorityBlockingQueue（优先级队列）

底层：二叉堆（数组实现）+ ReentrantLock

```java
// 创建
PriorityBlockingQueue queue = new PriorityBlockingQueue();        // 默认容量11
PriorityBlockingQueue queue = new PriorityBlockingQueue(100);     // 指定初始容量
PriorityBlockingQueue queue = new PriorityBlockingQueue(100, Comparator.comparingInt(o -> o));

// 示例：任务优先级队列
class Task implements Comparable<Task> {
    int priority;
    String name;
    
    @Override
    public int compareTo(Task o) {
        return Integer.compare(o.priority, this.priority);  // 优先级高的在前
    }
}

PriorityBlockingQueue<Task> queue = new PriorityBlockingQueue<>();
queue.put(new Task(1, "普通任务"));
queue.put(new Task(10, "紧急任务"));
queue.put(new Task(5, "重要任务"));

Task task = queue.take();  // 取出优先级最高的：紧急任务
```

特点：

*   按优先级取出元素（优先级高的先出）
*   不保证同优先级元素的顺序

### DelayQueue（延迟队列）

底层：PriorityBlockingQueue（内部元素需实现 Delayed 接口）

```java
// 创建
DelayQueue<DelayedElement> queue = new DelayQueue<>();

// 示例：订单超时取消
class Order implements Delayed {
    long expireTime;
    String orderId;
    
    public Order(String orderId, long delayMs) {
        this.orderId = orderId;
        this.expireTime = System.currentTimeMillis() + delayMs;
    }
    
    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(expireTime - System.currentTimeMillis(), TimeUnit.MILLISECONDS);
    }
    
    @Override
    public int compareTo(Delayed o) {
        return Long.compare(this.expireTime, ((Order) o).expireTime);
    }
}

DelayQueue<Order> queue = new DelayQueue<>();
queue.put(new Order("ORDER-001", 30000));  // 30秒后过期

// 消费者
while (true) {
    Order order = queue.take();  // 阻塞直到延迟期满
    System.out.println("取消订单: " + order.orderId);
}
```

特点：

*   只有延迟期满的元素才能被 take 取出
*   队首元素是延迟最短的元素

适用场景：

*   定时任务调度
*   缓存过期清理
*   订单超时取消

### LinkedTransferQueue（传输队列）

```java
// 创建
LinkedTransferQueue<String> queue = new LinkedTransferQueue<>(); // 无界

// 生产者（等待消费者取走）
new Thread(() -> {
    try {
        queue.transfer("必须被消费的消息");  // 阻塞直到被 take
        System.out.println("消息已被消费");
    } catch (InterruptedException e) {}
}).start();

// 消费者
new Thread(() -> {
    try {
        Thread.sleep(1000);
        String msg = queue.take();
        System.out.println("消费: " + msg);
    } catch (InterruptedException e) {}
}).start();
```

特点：

*   相比 SynchronousQueue 更灵活，支持有界语义
*   `transfer()`：生产者阻塞直到被消费者取走
*   `tryTransfer()`：非阻塞版本

适用场景：

*   高性能生产者-消费者
*   需要确认消息被消费的场景

### LinkedBlockingDeque（双端队列）

底层：双向链表 + ReentrantLock

```java
// 创建
LinkedBlockingDeque deque = new LinkedBlockingDeque();      // 无界
LinkedBlockingDeque deque = new LinkedBlockingDeque(100);   // 有界

// 双端操作
deque.putFirst("头部元素");
deque.putLast("尾部元素");

String first = deque.takeFirst();
String last = deque.takeLast();
```

特点：

*   可在队列两端操作
*   支持工作窃取（Work Stealing）

适用场景：

*   双端操作场景
*   工作窃取算法（ForkJoinPool 使用）

## 拒绝策略

线程池的拒绝策略（RejectedExecutionHandler）是在**线程池无法接受新任务时**触发的保护机制，主要有以下 4 种内置策略。

### 一、四种内置拒绝策略对比

通过实现 `RejectedExecutionHandler` 接口，Java 提供了四种内置策略，可通过 `Executors` 工厂类或 `ThreadPoolExecutor` 构造方法直接使用：

| 策略                    | 类名                                     | 行为                                                         | 适用场景                                                     |
| :---------------------- | :--------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **AbortPolicy（默认）** | `ThreadPoolExecutor.AbortPolicy`         | **直接抛出 `RejectedExecutionException`**。需要上层代码捕获异常并处理，是默认策略。 | 任务提交不容失败、需要明确感知系统负载的核心业务。           |
| **CallerRunsPolicy**    | `ThreadPoolExecutor.CallerRunsPolicy`    | **由调用者线程（提交任务的线程）直接执行该任务**。例如，主线程会自己跑这个任务，从而降低任务提交速度。 | 追求系统稳定性，允许任务执行延迟，不希望对任务造成损失的非核心场景。 |
| **DiscardPolicy**       | `ThreadPoolExecutor.DiscardPolicy`       | **静默丢弃任务，不抛异常，不记录日志**。任务被直接丢弃，调用方完全无感知。 | 对数据一致性要求极低、允许丢数据的场景（如日志记录、打点上报）。 |
| **DiscardOldestPolicy** | `ThreadPoolExecutor.DiscardOldestPolicy` | **丢弃任务队列中最老的（等待时间最长的）未执行任务**，然后尝试再次提交当前任务。 | 新任务优先级高于旧任务，或旧任务已无意义的场景（如实时行情刷新）。 |

### 二、拒绝策略触发条件

当线程池调用 `submit()` 或 `execute()` 提交任务时，如果遇到以下任意一种情况，就会触发拒绝策略：

1. **线程池已关闭**：调用了 `shutdown()` 或 `shutdownNow()`，不再接收新任务。
2. **任务队列已满且线程数已达上限**：当：
   - 当前线程数 ≥ `corePoolSize`（核心线程数）
   - 任务队列已满（`workQueue.offer()` 返回 `false`）
   - 当前线程数 ≥ `maximumPoolSize`（最大线程数）
     此时线程池已满负荷运转，无法处理新任务。

> **注意**：如果队列是无界队列（如 `LinkedBlockingQueue` 无容量限制），永远不会满，因此永远不会触发拒绝策略。这可能导致内存耗尽。

### 三、使用示例

```java
import java.util.concurrent.*;

public class RejectTest {
    public static void main(String[] args) {
        // 核心线程数 2，最大线程数 4，队列容量 2
        BlockingQueue<Runnable> queue = new ArrayBlockingQueue<>(2);
        
        ThreadPoolExecutor executor = new ThreadPoolExecutor(
            2, 4, 60L, TimeUnit.SECONDS, queue,
            new ThreadPoolExecutor.CallerRunsPolicy()  // 设置拒绝策略
        );
        
        // 模拟提交任务（此时会触发拒绝策略）
        for (int i = 1; i <= 10; i++) {
            final int taskId = i;
            executor.execute(() -> {
                System.out.println(Thread.currentThread().getName() + " 执行任务 " + taskId);
                try { Thread.sleep(100); } catch (InterruptedException e) {}
            });
        }
        
        executor.shutdown();
    }
}
```

### 五、最佳实践建议

| 业务场景                 | 推荐策略              | 原因                             |
| :----------------------- | :-------------------- | :------------------------------- |
| **核心交易、支付**       | `AbortPolicy`         | 必须明确感知失败，人工介入或补偿 |
| **流量削峰、异步处理**   | `CallerRunsPolicy`    | 减慢任务提交速度，保护系统稳定   |
| **日志上报、非关键统计** | `DiscardPolicy`       | 允许丢失，不影响主业务           |
| **实时行情、消息推送**   | `DiscardOldestPolicy` | 新数据覆盖旧数据，保持最新状态   |
| **高可靠业务**           | **自定义**            | 持久化 + 定时重试，确保最终执行  |

可以用一个比喻来记忆：想象一个接待窗口（线程池）：

- **AbortPolicy**：窗口爆满时，直接喊“不干了！”，并报警（抛异常）；
- **CallerRunsPolicy**：窗口爆满，让来办事的人（调用者）自己去后台处理；
- **DiscardPolicy**：窗口爆满，新来的人直接无视，当没发生过；
- **DiscardOldestPolicy**：窗口爆满，把队伍里站最久的人赶走，给新人腾位置。

## 工作线程的扩缩容

- **核心线程 (core threads)**：线程池的“基本盘”，默认会一直存活（除非设置了`allowCoreThreadTimeOut`）。
- **非核心线程 (extra threads)**：在核心线程都忙且队列满后，临时创建的“援军”，空闲一段时间后会被销毁。

还有一个重要特性：线程池的“懒启动”。默认情况下，线程池在创建后，里面是**没有任何线程**的。线程的创建是**随着任务的提交**才逐步发生的。

### 线程数自动增加的过程

线程数的增加，是任务不断提交、线程池状态发生变化的被动过程。以 `ThreadPoolExecutor.execute()` 方法的核心逻辑为例，它遵循以下步骤：

1. **当第一个任务提交时**：如果当前工作线程数 `workerCount` < `corePoolSize`，会立刻创建一个新线程来执行这个任务。
2. **当后续任务继续提交时**：
   - 只要 `workerCount` < `corePoolSize`，线程池就会倾向于创建新线程来处理任务，而不是把任务放队列。
   - 当 `workerCount` 达到 `corePoolSize` 后，新提交的任务会被放入**阻塞队列**（`BlockingQueue`）。
3. **当队列也满了之后**：如果队列已满，且 `workerCount` < `maximumPoolSize`，此时线程池会“破例”创建新的非核心线程来加速处理任务。这就是线程数的**自动扩容**。
4. **当 `workerCount` 达到 `maximumPoolSize` 且队列也满了**：新任务会触发拒绝策略。

**用一个例子简单描述这个过程：**

假设 `corePoolSize = 2`，`maximumPoolSize = 5`，`workQueue` 容量为 3。

- **任务1到来**：`workerCount=0 < 2`，**创建线程A执行**。
- **任务2到来**：`workerCount=1 < 2`，**创建线程B执行**。
- **任务3、4、5到来**：`workerCount=2 = corePoolSize`，不创建新线程，任务**进入队列**（队列变为 `[3, 4, 5]`）。
- **任务6到来**：队列已满，且 `workerCount=2 < 5`，**创建线程C执行**（非核心线程）。
- **任务7到来**：队列已满，`workerCount=3 < 5`，**创建线程D执行**。
- **任务8到来**：`workerCount=5 = maximumPoolSize`，队列已满，触发拒绝策略。

### 线程数自动减少的过程

线程数的减少，依赖于**非核心线程的空闲超时机制**。

- 当非核心线程执行完当前任务后，它不会立即销毁，而是会尝试从队列中获取新任务。
- 它会在一个**时间周期**（由 `keepAliveTime` 和 `unit` 定义）内，循环等待新任务。如果在这个周期结束后仍未获取到新任务，这个线程就会被自动销毁。
- 核心线程默认不受这个超时机制影响。但如果你调用了 `allowCoreThreadTimeOut(true)`，核心线程在空闲时也会被回收，缩容会更彻底。

**延续前面的例子：**

当高峰过去，假设任务5、6、7、8都执行完毕，队列也被清空。此时：

- 线程A和B（核心线程）会继续存活，等待新任务。
- 线程C、D、E（非核心线程）在等待任务超过 `keepAliveTime` 时间后，**会被逐个回收销毁**。最终工作线程数会回落到 `corePoolSize` 的大小。
