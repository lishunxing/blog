# 持久化

Redis 主要提供 两种 持久化方案：RDB 和 AOF，以及 Redis 4.0 之后推出的 混合持久化。

## RDB (Redis DataBase) —— 快照方式

原理：在指定时间间隔内，将内存中的数据集的快照以二进制文件（dump.rdb）写入磁盘。

触发方式：

*   自动触发：配置 `save <seconds> <changes>`，如 `save 900 1`
*   手动触发：`save`（阻塞，不推荐生产使用）、`bgsave`（fork 子进程，推荐）

> 在 900 秒（15 分钟）内，如果至少有 1 个 key 被修改，就自动执行一次 RDB 快照。

几个常见的默认配置

Redis 默认配置通常包含三条规则，满足任意一条就会触发快照：

```conf
save 900 1     # 15分钟内，至少有1个key变化
save 300 10    # 5分钟内，至少有10个key变化
save 60 10000  # 1分钟内，至少有10000个key变化
```

Redis 会持续跟踪自上次快照以来，有多少个 key 被修改。每执行一次 bgsave 后，这个计数器就会重置为 0。

优点：

*   文件紧凑，体积小，适合备份和灾难恢复
*   恢复大数据集时速度比 AOF 快
*   对性能影响小（fork 子进程，主进程继续处理命令）

缺点：

*   实时性差，两次快照之间的数据可能丢失
*   数据集较大时，fork 子进程可能耗时较长，导致短暂停顿

## AOF (Append Only File) —— 日志方式

原理：记录每个写操作命令，追加到 appendonly.aof 文件，重启时重放命令恢复数据。

配置：appendonly yes

同步策略（`appendfsync`）：

*   `always`：每次写命令都同步，最安全但性能最低
*   `everysec`：每秒同步一次，兼顾性能与安全（推荐）
*   `no`：由操作系统决定，性能最好但数据可能丢失

优点：

*   数据安全性高（最多丢失 1 秒数据）
*   AOF 文件是纯文本格式，可读性强，便于理解和修复

缺点：

*   文件体积比 RDB 大
*   恢复速度比 RDB 慢
*   性能开销比 RDB 略高

### AOF 重写

AOF 重写（AOF Rewrite）可以理解为对 AOF 日志文件的一次“无损压缩”和“整理”。它的目的是解决 AOF 文件无限膨胀和恢复速度慢的问题。

简单来说，AOF 文件记录了每个写命令。随着服务运行，同一个 key 可能被修改成千上万次，导致 AOF 文件里存满了对同一份数据的无效历史命令。AOF 重写就是把这些冗长的历史命令，替换成一条能代表最终状态的最简命令。

#### AOF 重写的核心原理

AOF 重写的核心思想是：读取当前内存中的数据库状态，反向生成对应的写命令，写入一个新的 AOF 文件。

*   旧文件：记录的是“过去做了什么”。
*   重写过程：关注的是“现在是什么”。

重写流程（由子进程完成）：

1.  Redis 主进程 `fork` 出一个子进程。
2.  子进程读取当前内存中的所有 key 和 value。
3.  对于每个 key，子进程生成一条能创建当前状态的命令（比如 `SET` 或 `RPUSH` 等）。
4.  子进程将所有生成的命令写入一个新的 AOF 文件（`temp-rewrite.aof`）。
5.  在子进程重写期间，主进程仍然在处理新请求。它会将新收到的写命令同时写入 AOF 重写缓冲区。
6.  子进程完成后，主进程会将“AOF 重写缓冲区”中的增量命令也追加到新文件中。
7.  最后，主进程用这个全新的、体积很小的文件，原子地替换掉旧的、庞大的 AOF 文件。

#### AOF重写触发条件

*   自动触发：通过配置 `redis.conf` 中的条件来控制。

        # 当前 AOF 文件大小 比上次重写时的大小大100% 时触发
        auto-aof-rewrite-percentage 100
        # 且 AOF 文件至少达到 64MB 才触发
        auto-aof-rewrite-min-size 64mb

*   手动触发：连接 Redis 执行命令。

    ```bash
    redis-cli BGREWRITEAOF
    ```

*   AOF 重写：是 Redis 对 AOF 文件进行的一次后台整理，用最终状态替换历史命令。

*   核心价值：控制文件大小（节省磁盘）和加快恢复速度（启动时执行的命令少）。

*   重要特征：它是一个不依赖旧文件的独立过程，不阻塞主进程的正常服务。

## 混合持久化（Redis 4.0+）

原理：AOF 重写时，将 RDB 快照写在 AOF 文件开头，后续追加增量日志。

配置：`aof-use-rdb-preamble` yes

优点：结合 RDB 恢复快和 AOF 数据安全的特点。

缺点：文件可读性变差（混合格式）。

混合持久化生成的文件，在 Redis 的不同主要版本中，其具体形态有所不同

简单来说，它是一个 以RDB格式开头，接着是AOF格式命令的单一文件。你可以把它想象成一个由两部分“粘合”起来的文件：

文件头部（RDB格式）：这是一个二进制格式的数据快照，对应启动重写那一刻的完整数据集。

文件尾部（AOF格式）：这部分是纯文本格式的Redis命令，记录了从RDB快照生成之后到重写完成之前，所有新执行的写操作。

这样的设计让它在恢复数据时，可以快速加载RDB快照，再回放少量AOF命令即可，兼顾了速度和数据完整性。

### 开启配置

*   Redis 4.0：引入该功能，默认关闭（`no`）
*   Redis 5.0+：默认开启（`yes`）

```conf
# 1. 开启 AOF（这是前提）
appendonly yes

# 2. 开启混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes

# 验证是否开启
redis-cli CONFIG GET aof-use-rdb-preamble

```

> 开启混合持久化后，为避免重复存储和双份 fork 开销，建议注释掉	 save 规则
>
> ```conf
> # 注释掉或删除以下行
> # save 900 1
> # save 300 10
> # save 60 10000
> ```

## 生产建议

1.  推荐两者同时开启：既有 RDB 的快速恢复，又有 AOF 的数据安全
2.  从库可关闭持久化：仅主库做持久化，从库只做备份和读
3.  重要数据建议：`appendfsync everysec` + 定期 `bgsave`
4.  Redis 4.0+：开启混合持久化，获得最佳效果

# Redis集群

Redis 的集群配置主要分为三种模式，分别解决高可用、数据冗余和水平扩展的问题。下面是这三种模式的配置方法、适用场景和核心配置项。

## 三种集群模式概览

| 模式                 | 核心能力             | 适用场景                     | 复杂度 |
| :------------------- | :------------------- | :--------------------------- | :----- |
| **主从复制**         | 数据冗余、读写分离   | 读多写少的场景，需要数据备份 | 低     |
| **哨兵模式**         | 自动故障转移、高可用 | 生产环境，需要自动恢复       | 中     |
| **Cluster 分片集群** | 水平扩展、数据分片   | 海量数据，需要横向扩容       | 高     |

## 主从复制配置

### 配置方式

| 服务器      | 角色             | 端口 |
| :---------- | :--------------- | :--- |
| 192.168.1.2 | **Master（主）** | 6379 |
| 192.168.1.3 | **Slave（从）**  | 6379 |
| 192.168.1.4 | **Slave（从）**  | 6379 |

方式一：修改从节点的配置文件`/etc/redis/redis.conf`（永久生效）

```conf
# 在从节点的 redis.conf 中添加 主节点无需修改
replicaof <master_ip> <master_port>

# 基础配置（与主节点类似）
bind 0.0.0.0
port 6379
daemonize yes
pidfile /var/run/redis_6379.pid
logfile /var/log/redis_6379.log

# 持久化配置（从节点通常也开启 AOF，方便故障切换）
appendonly yes
appendfsync everysec

# 从节点核心配置：指定主节点
replicaof 192.168.1.2 6379

# 如果主节点有密码，必须配置
masterauth your_master_password

# 可选：设置从节点自己的密码（只影响客户端连接）
requirepass your_replica_password

# 可选：设置从节点为只读（默认就是只读）
replica-read-only yes
```

方式二：命令行临时生效（重启后失效）

```bash
redis-cli -p 6379
192.168.1.3:6379> REPLICAOF 192.168.1.2 6379
```

### 验证主从关系

```bash
# 主节点查看
redis-cli -h 192.168.1.2 -p 6379 -a your_master_password INFO replication

# 预期输出
# Replication
role:master
connected_slaves:2
slave0:ip=192.168.1.3,port=6379,state=online,offset=12345,lag=0
slave1:ip=192.168.1.4,port=6379,state=online,offset=12345,lag=0
```

可以看到角色（master/slave）和连接的从节点数量。

### 特点

*   主节点可读写，从节点只读
*   数据异步复制，可能存在短暂不一致
*   主节点故障不会自动切换，需人工介入

> Redis 引入主从复制，核心是为了解决单机版 Redis 的三大痛点：单点故障、读性能瓶颈和数据容灾。

### 单机Redis的问题

宕机即不可用：服务器一旦宕机，整个缓存系统崩溃，所有请求直接压垮后端数据库。

读压力过大：所有读写请求都压在这一台机器上，当读并发很高时（如秒杀查库存），CPU或网络带宽会成为瓶颈。

数据丢失风险：服务器硬盘损坏，所有缓存数据直接丢失，没有备份。

主从复制通过将数据同步到多台机器，把一份数据变成了多份，从而解决了上述问题。具体体现在以下四个方面：

#### 实现读写分离，分担读压力 (最主要原因)

*   主节点 (Master)：负责处理写操作（`SET`， `DEL`等）和关键的读操作。
*   从节点 (Slave/Replica)：专门负责处理大量的读操作（`GET`等）。
*   效果：你可以不停增加从节点来横向扩展读能力，大幅减轻主节点的负担。例如，一个高并发网站的首页数据，可以配置一主两从，将读请求分散。

#### 提供数据冗余，实现高可用 (HA)

主从复制是哨兵模式和集群模式的基础。

*   数据备份：从节点是主节点数据的实时热备份。即使主节点硬盘损坏，从节点上还有完整的数据。
*   故障转移基础：当主节点宕机时，哨兵可以自动将某个从节点升级为新的主节点。如果没有从节点，主节点宕机就只能等待人工恢复。

#### 支持故障恢复，避免单点故障

*   手动恢复：如果主节点宕机且没有自动故障转移，运维人员可以执行 `SLAVEOF` 命令，将一台从节点指向为新的主节点，系统能在几分钟内恢复服务。
*   避免雪崩：防止因缓存实例宕机导致数据库瞬间压力过大而崩溃。

#### 实现数据备份而不影响服务

*   如果你想备份 Redis 数据（例如每天全量备份到云存储），直接在主节点上执行 `BGSAVE` 会消耗性能。
*   最佳实践：在从节点上执行备份操作。从节点只负责读和备份，即使备份时短暂阻塞，也不会影响主节点的写入业务。

## 主从一致性原理

主从复制保证的并不是实时强一致性（因为会有延迟），而是最终一致性。

### 核心同步机制：复制积压缓冲区与偏移量

Redis 主从复制依赖三个核心概念来保证数据最终能同步到从节点：

*   复制积压缓冲区：主节点上的一块固定大小的内存队列（默认 1MB）。主节点每执行一条写命令，除了写入内存，还会往这个缓冲区和网络连接里各发一份。
*   复制偏移量：主节点每发送一个字节的数据，自己的偏移量就增加；从节点每收到一个字节的数据，自己的偏移量也增加。这个数字就像数据同步的“进度条”。
*   PSYNC 命令：当从节点断线重连后，它会带着自己上次同步到的偏移量去问主节点：“我断线了，我上次看到进度是 100，你继续从 101 发给我吧？”

关键点来了：如果主节点的复制积压缓冲区足够大，能存下从节点断线期间的所有命令，那么重连后只需要增量同步（部分重同步），瞬间就能追上。但如果缓冲区太小，存不下断线期间的命令，那就会触发一次全量同步（RDB 生成 + 传输 + 加载），耗时较长。

### 为什么是“最终一致”而不是“强一致”

因为在异步复制下，从节点一定落后于主节点。这个落后可能只有几毫秒，也可能达到几秒（取决于网络延迟和从节点负载）。

*   主节点写入成功，从节点还未同步：这是最常见的状态。如果此时主节点突然宕机，这部分数据就丢失了（因为还没发给从节点），这就违反了强一致性。
*   客户端从从节点读取：如果客户端写数据到主节点后，立刻去从节点读取刚写入的数据，很可能读不到。这就是主从延迟问题。

### 如何尽量保证（或确认）一致性？

虽然默认是异步，但 Redis 也提供了一些机制，让应用层可以选择牺牲部分性能来换取更高的一致性。

对写操作：要求同步复制 WAIT  Redis 提供了 WAIT 命令。

客户端在写入主节点后，可以执行：

```bash
127.0.0.1:6379> SET key1 value1
OK
127.0.0.1:6379> WAIT 1 5000
(integer) 1
```

这条 WAIT 1 5000 的意思是：等待这个写操作被至少 1 个从节点确认收到，最长等待 5000 毫秒。

*   返回值：表示实际确认收到这条命令的从节点数量。
*   效果：只有当至少 1 个从节点也存了这条数据后，主节点才会继续处理下一条客户端命令。这能大大降低主从切换时的数据丢失风险。

注意：`WAIT` 不保证从节点已经执行了命令，只保证从节点收到了（存在内核缓冲区或 AOF 缓冲区）。要想更严格，还需要配置从节点的 `repl-diskless-sync` 等参数。

### 配置建议

*   首次连接：一定会发生一次全量同步，这是不可避免的。
*   网络抖动：增大 `repl-backlog-size`（例如设置为 100MB-1GB），可以有效避免因短暂断网导致的频繁全量同步。
*   从节点过多：如果有太多从节点同时发起全量同步，会大量消耗主节点的 CPU 和内存资源。建议采用级联复制，即让某些从节点再作为其他从节点的主节点，分担主节点的同步压力。
*   数据安全：主从复制默认是异步的，主节点返回成功时并不保证从节点已经收到数据。如果需要更强的数据安全保证，可以配合使用 `WAIT` 命令或配置 `min-replicas-to-write`。

### JAVA中使用Redission保证一致性读取

假设有需要业务强一致性的读取, 在主节点修改了数据后从节点必须获取到最新的修改数据, redisson也提供了方法

#### 方法一：使用 get(...) 方法（推荐）

Redisson 为 RMap、RBucket 等对象提供了 get(...) 的重载方法，可以临时指定读取模式

```java
// 从主节点强制读取一次最新数据
RBucket<String> bucket = redissonClient.getBucket("user:status:123");
String value = bucket.get(ReadMode.MASTER);

// 或者在操作 RMap 时指定
RMap<String, String> map = redissonClient.getMap("user:data:123");
String password = map.get("password", ReadMode.MASTER);
```

#### 方法二：直接使用主节点地址创建客户端

如果部分代码逻辑非常依赖强一致性，也可以单独创建一个只连接主节点的 RedissonClient 对象，专门用于处理关键数据。

```java
// 创建一个专门连接主节点的客户端
Config masterConfig = new Config();
masterConfig.useSingleServer().setAddress("redis://your-master-ip:6379");
RedissonClient masterRedissonClient = Redisson.create(masterConfig);

// 在需要的地方注入并使用
masterRedissonClient.getBucket("critical:data").get();
```

## 哨兵模式

主从复制解决了“数据冗余”和“读性能”，但没有解决“故障自动恢复”的问题。

如果把主从复制比作一个备用轮胎（爆胎了可以手动换上），那哨兵模式就是一个自动换胎的机器人（爆胎了自动换好，业务几乎无感知）。

### 哨兵模式做了什么

哨兵模式在主从复制的基础上，增加了一个监控和自动调度层——哨兵（Sentinel）。它专门干三件事：

| 功能             | 说明                                                         |
| :--------------- | :----------------------------------------------------------- |
| **监控**         | 哨兵会不断 ping 你的主节点和从节点，问它们“你还活着吗？”     |
| **通知**         | 如果某个节点挂了，哨兵可以发送告警给管理员。                 |
| **自动故障转移** | 如果主节点挂了，哨兵会自己开会（Raft 共识算法）选出一个领头哨兵，然后由它负责：从从节点里选一个当新主，修改其他从节点的配置指向新主，通知客户端新的主地址。 |

### 配置哨兵（Sentinel）

哨兵的配置文件通常是 sentinel.conf。按照下面的步骤，在三台服务器上分别配置哨兵（为了方便，可以先在一台上配置好，再复制到其他服务器）

### 第一步: 每台服务器上都要创建一个哨兵配置文件。

推荐路径：/usr/local/redis/sentinel.conf。

### 第二步：编辑配置文件&#x20;

在 sentinel.conf 文件中，添加以下核心配置。这是最关键的一步，一定要理解每个参数的含义

```conf
# 1. 基础网络配置
bind 0.0.0.0                              # 允许所有IP连接，生产环境可绑定具体内网IP
port 26379                                 # 哨兵默认端口
daemonize yes                              # 后台运行
protected-mode no                          # 关闭保护模式，允许外部连接（如果配置了密码则可不关）

# 2. 核心监控配置
# 格式: sentinel monitor <主节点名称> <主节点IP> <主节点端口> <quorum>
sentinel monitor mymaster 192.168.1.2 6379 2

# 3. 安全认证（如果你的Redis设置了密码，这两项必须配置！）
sentinel auth-pass mymaster your_redis_password
# 若主从密码不同，这里配置的是所有节点的统一密码

# 4. 故障转移相关配置
sentinel down-after-milliseconds mymaster 5000     # 主观下线时间（5秒无响应即认为可疑）
sentinel failover-timeout mymaster 60000           # 故障转移超时时间（60秒）
sentinel parallel-syncs mymaster 1                 # 故障转移后，同时从新主同步数据的从节点数
```

### 第三步: 启动与验证

在三台服务器上依次执行以下命令，启动哨兵进程

```bash
redis-sentinel /usr/local/redis/sentinel.conf
```

验证哨兵状态

### 哨兵模式的故障转移

哨兵模式的故障转移，核心可以理解为 “监控、投票、换主、改配” 这四个步骤的自动化。整个过程由哨兵集群自主完成，对客户端而言，就像一次自动的故障切换

#### 第一步：主观下线 (SDOWN) - 发现异常

这是一个哨兵的初步判断。每个哨兵会以每秒一次的频率，向它监控的节点（主、从节点）发送 PING 命令。

*   正常：节点返回 `PONG` 或 `LOADING` 等正常响应。
*   异常：如果节点在 `down-after-milliseconds` 设定的时间内（例如 5 秒）都没有响应，那么这个哨兵就会主观地认为这个节点挂了，并标记为 `sdown` (Subjectively Down)。

> 注意：这只是单个哨兵的个人看法，可能是网络抖动造成的误判。

#### 第二步：客观下线 (ODOWN) - 达成共识

当某个哨兵（我们称它为 Sentinel-A）发现主节点为 sdown 后，它不会立即动手切换，而是会去询问其他哨兵：“嘿，你们觉得主节点还活着吗？”

*   Sentinel-A 向其他哨兵发送 `SENTINEL is-master-down-by-addr` 命令。
*   其他哨兵根据自己与主节点的连接情况，回复投票。
*   如果 `quorum` 数量（在哨兵配置文件中设置）的哨兵都认为主节点已 down，那么 Sentinel-A 就会将主节点标记为 `odown` (Objectively Down)，客观下线。

> quorum 的作用：这是启动故障转移的最低门票。例如，3 个哨兵里设 quorum=2，那么只要 2 个哨兵确认主节点挂了，就基本可以认定是真的故障，而不是网络问题。

#### 第三步：领导者选举 - 选出执行者

主节点被确认为 `odown` 后，故障转移只能由一个哨兵来执行。那么，哪个哨兵来执行呢？哨兵们会通过 Raft 算法 进行一场快速选举，选出一个 Leader。

*   竞选资格：通常，第一个发现 `odown` 的哨兵会发起竞选。
*   投票规则：每个哨兵（包括自己）只能投一票。只有获得 半数以上 哨兵投票的候选者才能成为 Leader。
*   结果：如果集群有 3 个哨兵，一个候选者需要至少 2 票才能成为 Leader。这也是为什么哨兵集群推荐部署为奇数个。

#### 第四步：故障转移 - 核心切换逻辑

被选出来的 Leader 哨兵 会开始执行故障转移，主要分三步：

1.  挑选一个合适的从节点作为新主节点\
    Leader 哨兵会从挂掉主节点的所有从节点里，按照一套优先级规则选出一个“最合适的”：

    *   过滤：剔除断线或网络连不上的从节点。
    *   优先级：优先选择 `slave-priority`（或 `replica-priority`）配置值最小的（数字越小优先级越高）。
    *   数据最完整：如果优先级相同，选择复制偏移量（`slave_repl_offset`）最大、数据最新的那个。
    *   ID最小：如果上面都相同，选择 run-id 最小的。
2.  晋升新主节点\
    Leader 哨兵向选中的从节点发送 `SLAVEOF NO ONE` 命令，让它成为新的主节点。随后通过 `INFO` 命令确认其角色已变更为 `master`。
3.  改造其他从节点\
    Leader 哨兵向其他从节点发送 `SLAVEOF <新主IP> <新主端口>` 命令，让它们结束对旧主的复制，转而复制新的主节点。

#### 第五步：更新配置并通知客户端

*   更新哨兵配置：Leader 哨兵将新的主节点信息广播给所有哨兵，大家各自更新自己的 `sentinel.conf` 配置文件，将监控的主节点改为新的地址。
*   通知客户端：客户端通过订阅哨兵，会收到 `+switch-master` 事件，得知新主节点的地址，并重新建立连接。后续的写请求将全部指向新的主节点。

#### 对旧主节点的处理

*   原来的主节点恢复上线后，它会发现自己已经有了新的“主人”（当前的主节点）。
*   它会自动降级，成为一个从节点，并开始从新的主节点同步数据，补齐它“离线”期间丢失的所有写操作。
*   在此期间, 可能会发生脑裂现象。

### 哨兵模式脑裂问题怎么解决

脑裂（Split-Brain）的本质是：**网络分区导致旧主节点与集群失联，但未自我停服，仍在接收客户端写入**；而集群另一侧已选出新主节点。 当网络恢复，旧主被迫降级为从节点，其“孤立期间”写入的数据会被清空，造成数据丢失。

Redis 脑裂问题没有 100% 完美的解决方案，但可以通过 “拒绝不可靠写入” 和 “确保仲裁共识” 的组合策略，将数据丢失风险降到最低。

#### 方案一：通过 min-replicas-to-write 限制写入（最核心配置)

这是 Redis 官方提供的标准解决方案。通过在 旧主节点 的 redis.conf 中配置以下两个参数，可以让它在与足够多的从节点失去联系时，自动拒绝写请求，从而阻断“脏数据”的产生

```conf
# 必须至少有 1 个从节点连接正常
min-replicas-to-write 1
# 从节点延迟不能超过 10 秒
min-replicas-max-lag 10
```

配置原理与效果

1.  条件限制：只有当从节点数量 `>= min-replicas-to-write` 且 这些从节点的延迟 `<= min-replicas-max-lag` 时，主节点才接受写操作。
2.  触发保护：一旦发生网络分区，旧主节点与从节点失联，`min-replicas-to-write` 条件不再满足。此时，针对该节点的所有写请求都将被拒绝，客户端会收到 `(error) NOREPLICAS Not enough good replicas to write.` 的错误。
3.  效果：由于旧主节点在隔离期间没有写入新数据，当网络恢复后，它被降级并同步新主节点数据时，不会有任何业务数据丢失

配置建议：

*   `min-replicas-to-write` 设置为 `1` 即可，表示只要有至少一个从节点能同步，系统就正常工作。
*   `min-replicas-max-lag` 建议不要设得太小（如 `1` 或 `2`），否则网络短暂抖动可能误触发保护机制，导致服务不可用。`5` 或 `10` 秒是相对稳妥的值。

当你的网络稳定，至少有一个从节点正常运行且延迟在 `min-replicas-max-lag`（例如默认10秒）以内时，`min-replicas-to-write 1` 几乎不会产生额外的性能开销。

*   主节点只是做了一次计数检查：它在处理每个写命令前，会快速检查一下当前有多少个符合条件的从节点在线 （主节点自己维护在内存的从节点列表， 从节点会每秒向自己上报信息，每次上报都会更新 `lag`( 延迟秒数)值, 超过了 `min-replicas-max-lag` 的值才会被判定为不健康的数量）。
*   不等待从节点响应：这个检查是即时的，并不会像 `WAIT` 命令那样阻塞等待从节点确认收到数据。因此，对命令执行耗时的影响通常在微秒级别，可以忽略不计

### 方案二：哨兵模式下的仲裁配置

在哨兵模式下，除了限制写入，还需要确保哨兵集群能正确决策，避免因误判而发起不必要的故障转移。

#### 合理配置 quorum

在 sentinel.conf 中，`quorum` 参数决定了至少需要多少个哨兵同意，才能判定主节点客观下线并启动切换

```conf
sentinel monitor mymaster 192.168.1.2 6379 2
```

*   建议：`quorum` 应配置为哨兵总数的大多数。例如，3 个哨兵时 `quorum` 设为 2；5 个时设为 3。
*   作用：这可以防止因单个哨兵误判或网络抖动而触发不必要的切换。

#### 部署奇数个哨兵

哨兵集群的选举依赖于多数派投票机制。3 个或 5 个哨兵是生产环境的标配。偶数个哨兵（如 2 个）在网络分区时，可能双方都无法获得超过半数的选票，导致服务瘫痪。

### 方案三：集群模式（Redis Cluster）的天然优势

如果你使用的是 Redis Cluster（分片集群），它相比“主从+哨兵”架构，对脑裂有更强的抵抗力

*   多数派决策：Cluster 的故障判定基于 “大多数主节点” 的共识。一个孤立的分区如果只包含少数节点，无法选出新主，也无法进行写入。
*   epoch （纪元）机制：每个主节点都有一个唯一的、递增的 epoch 值，相当于它的“任期”。该机制类似于 Raft 算法，确保只有拥有最新、最高 epoch 的节点才能赢得选举，有效防止双主现

## Cluster 分片集群配置

主从+哨兵解决了“高可用”，但没有解决“海量存储和写性能的横向扩展”问题。

*   主从+哨兵：相当于给一辆卡车（主节点）配了多个备用轮胎（从节点）和自动换胎机器人（哨兵）。卡车爆胎能自动换上备胎继续开，但卡车的载货量没变。
*   Cluster：相当于直接把多辆卡车编成了一支车队。每辆车只装一部分货，同时每辆车也有自己的备用轮胎和换胎机器人。

当数据量太大（比如超过几百GB），或者写入并发太高（单台Redis扛不住）时，主从+哨兵的“单主节点”架构就会成为瓶颈，这时就需要Cluster

### 主从+哨兵架构的核心瓶颈

在“一主多从+哨兵”的架构里，无论如何增加从节点，世界上有且只有一个主节点。这会导致两个无法绕过的问题：

1.  单点写入瓶颈：所有的写操作（`SET`、`DEL`、`INCR`）都必须去主节点。如果业务写入量极大（比如每秒几万次、十几万次），主节点的CPU和网络带宽很快就会成为天花板。从节点加得再多，也只能分担读压力，对写压力毫无帮助。
2.  数据存储受限：一台Redis服务器的内存是有限的。为了性能，Redis的数据库基本都存储在内存中，假设服务器只有256GB内存，整个Redis集群的数据总量就不能超过256GB。当你的业务数据达到TB级别时，这个架构就无法支撑了。

### Cluster如何解决这两个问题

Redis Cluster引入了分片的概念，将数据分散到多个主节点上。

*   解决写入瓶颈：Cluster有多个主节点（例如3个主节点）。不同的写请求可以发给不同的主节点处理。总写入能力 ≈ 主节点数 × 单机写入能力。集群的写性能可以随着加入更多主节点而水平扩展。
*   解决存储限制：数据被打散存储在多个主节点上。例如你有3个主节点，总数据量就可以达到 `256GB * 3 = 768GB`。理论上，只要增加节点，就可以存储近乎无限的数据。

| 架构模式     | 核心能力             | 写性能扩展     | 存储容量扩展       | 高可用       | 适用场景                               |
| :----------- | :------------------- | :------------- | :----------------- | :----------- | :------------------------------------- |
| **主从复制** | 数据冗余、读写分离   | ❌ 无法扩展     | ❌ 无法扩展         | ❌ 需人工介入 | 小型应用、开发测试                     |
| **哨兵模式** | 自动故障转移，高可用 | ❌ 仍受限于单主 | ❌ 仍受限于单机内存 | ✅ 自动切换   | 中等规模、对高可用要求高、数据量<100GB |
| **Cluster**  | 分片、水平扩展       | ✅ 可线性扩展   | ✅ 可线性扩展       | ✅ 内部实现   | 大规模、高吞吐、数据量超过单机内存     |

### Cluster集群配置

配置 Redis Cluster 的核心思路是将多个 Redis 实例组织起来，让它们自动分担数据并互相备份。生产环境中最经典的配置就是 6 个节点（3 主 3 从）。

下面是一套通用的配置流程，可以适应你的网络环境（如 192.168.1.2 到 192.168.1.7）。

#### 准备工作

在动手之前，有两个关键概念需要了解，它们直接影响配置成败：

*   两大端口：每个集群节点需要开放两个 TCP 端口。

    *   数据端口：例如 `6379`，用于服务客户端。
    *   集群总线端口：例如 `16379`（通常是数据端口 + 10000），用于节点间内部通信（心跳、数据迁移）。防火墙必须同时开放这两个端口。
*   最少节点数：为保证高可用，集群至少需要 3 个主节点。每个主节点至少配 1 个从节点，因此 6 个节点是最小生产配置。

#### 详细配置步骤

| 节点  | IP 地址     | 数据端口 | 集群总线端口 | 角色   |
| :---- | :---------- | :------- | :----------- | :----- |
| 节点1 | 192.168.1.2 | 6379     | 16379        | 主节点 |
| 节点2 | 192.168.1.3 | 6379     | 16379        | 主节点 |
| 节点3 | 192.168.1.4 | 6379     | 16379        | 主节点 |
| 节点4 | 192.168.1.5 | 6379     | 16379        | 从节点 |
| 节点5 | 192.168.1.6 | 6379     | 16379        | 从节点 |
| 节点6 | 192.168.1.7 | 6379     | 16379        | 从节点 |

#### 修改 Redis 配置文件 (redis.conf)

在每一个节点上，修改 redis.conf 文件，启用集群模式。以下是核心配置项

```conf
# 基础网络配置
bind 0.0.0.0                               # 允许外部访问
port 6379                                  # 数据端口，请根据规划修改
cluster-enabled yes                        # 【关键】开启集群模式
cluster-config-file nodes-6379.conf        # 集群状态文件，自动生成
cluster-node-timeout 5000                  # 节点超时时间(毫秒)，超过则认为故障
cluster-announce-ip 192.168.1.2            # 【关键】节点实际IP，云环境/Docker必须配置
cluster-announce-port 6379                 # 节点数据端口
cluster-announce-bus-port 16379            # 节点总线端口

# 持久化（推荐开启）
appendonly yes                             # 开启AOF持久化，保障数据安全

# 后台运行
daemonize yes
```

> 注意：cluster-announce-\* 配置项在云服务、Docker 或 NAT 环境中至关重要，必须填写实际可访问的 IP。

#### 创建集群 (将节点关联起来)

在任意一台节点上（如 192.168.1.2），使用 redis-cli 执行创建命令。

```bash
# 格式：redis-cli --cluster create <节点1> <节点2> ... <节点N> --cluster-replicas <副本数>
redis-cli --cluster create \
  192.168.1.2:6379 \
  192.168.1.3:6379 \
  192.168.1.4:6379 \
  192.168.1.5:6379 \
  192.168.1.6:6379 \
  192.168.1.7:6379 \
  --cluster-replicas 1
```

*   `--cluster-replicas 1`：指定每个主节点配备 1 个从节点。
*   执行后，系统会自动将前 3 个节点设为主节点，后 3 个设为从节点，并分配 16384 个哈希槽。输入 `yes` 确认即可完成。

##### 分配 16384 哈希槽的原因

> 结论: 控制心跳包大小

Redis 集群中的节点需要互相发送心跳包（PING/PONG/MEET）来交换彼此的状态信息，包括节点是否存活、集群配置信息，以及非常重要的——每个节点负责的哈希槽信息。

一个 Redis 集群最多能容纳 16384 个主节点（每个节点至少一个槽）。这个上限在可预见的未来是足够的。

*   心跳包的数据结构：每个节点会用一个位序列（bitmap）来表示自己负责哪些哈希槽。如果哈希槽是 `n` 个，那么这个位序列的长度就是 `n` 位。

    *   `n = 16384`：位序列大小 = `16384 bits = 2048 bytes = 2KB`
    *   `n = 65535`：位序列大小 = `65535 bits ≈ 8KB`

*   网络开销放大效应：对于一个有 1000 个节点的集群，每个节点每秒都会收到和处理来自其他节点的多个心跳包。如果每个心跳包的数据量从 2KB 增加到 8KB，网络带宽的消耗将翻四倍，这会成为集群规模扩展的一个显著瓶颈。

*   官方作者的观点：Redis 作者 `@antirez` 在 GitHub 上曾对这个问题有过经典的回复。他明确指出，用 2KB 的位序列来表示所有槽位，对于普通集群已经足够；而使用 8KB 的位序列来支持 65535 个槽位，对于网络流量的压力太大。16384 是权衡后的最优选择。

#### 验证集群状态

```bash
# -c 表示集群模式，info 查看状态
redis-cli -c -h 192.168.1.2 -p 6379 cluster info
# 检查节点状态和槽位分配
redis-cli -c -h 192.168.1.2 -p 6379 cluster nodes
```

如果 cluster\_state 为 ok，且看到 master 和 slave 角色，说明集群已正常工作。

### 日常运维

*   扩容：启动新节点后，使用 `redis-cli --cluster add-node` 加入，再用 `reshard` 迁移数据槽。
*   故障转移：集群有自动故障转移能力。如果主节点挂了，它的从节点会自动晋升。主节点恢复后，会自动变成新主节点的从节点。
*   数据安全：集群默认是异步复制，在主节点故障且数据未同步到从节点时，可能丢失少量数据。如果对一致性要求极高，可以使用 `WAIT` 命令

### Cluster 是如何自己完成故障转移的

当你启动一个 Cluster 集群时，每个节点内部都同时运行着一个 “哨兵逻辑”，它们通过内部总线（集群总线端口）进行通信，共同完成监控和选举。整个过程对运维人员是透明的。

1.  监控：集群中的每个节点都会通过内部总线，周期性地给其他节点发送 `PING` 包来检查对方是否存活。
2.  主观下线：如果节点 A 发现节点 B 在 `cluster-node-timeout` 时间内没有响应，节点 A 就会标记节点 B 为 PFAIL (Possible Fail，主观下线)。
3.  客观下线：当一个主节点被大多数（超过半数）的其他主节点标记为 `PFAIL` 时，它就会被标记为 FAIL (客观下线)。
4.  故障转移：一旦主节点被标记为 `FAIL`，其下属的一个从节点就会发起选举。如果它获得了大多数主节点的投票，就会自动晋升为新的主节点，完成故障转移。

整个流程和哨兵模式非常像，区别在于：哨兵模式是哨兵进程在投票，而 Cluster 是主节点之间在投票。
