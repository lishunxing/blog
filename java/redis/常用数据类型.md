# 常用数据类型

Redis 的常用数据类型主要有五种：String、Hash、List、Set、Sorted Set。此外还有 Bitmap、HyperLogLog、GEO、Stream 等扩展类型。

## String（字符串）

### 特性

*   最基础的类型，值可以是字符串、数字（整数/浮点）、二进制数据（如图片）
*   最大容量 512MB
*   支持原子递增/递减操作

### 常用命令

```bash
SET key value
GET key
INCR key      # 原子+1
DECR key      # 原子-1
MSET k1 v1 k2 v2   # 批量设置
```

### 使用场景

| 场景            | 示例                                                    |
| :------------ | :---------------------------------------------------- |
| **缓存对象**      | 缓存用户信息：`SET user:1234 '{"name":"张三"}'`                |
| **分布式锁**      | `SET lock:order:1234 uuid NX EX 30`                   |
| **计数器**       | 文章阅读数：`INCR article:read:1001`  点赞数, 库存, 短信验证, 评论都可以做 |
| **Session共享** | 分布式系统中存储用户会话 token                                    |
| **全局ID生成**    | `INCR global:id` 获取唯一自增ID                             |
| **限流**        | 滑动窗口限流：`INCR user:rate:1234` + `EXPIRE`               |

## Hash（哈希表）

适合存储对象的属性，支持单独修改某个字段，非常节省内存。

### 特性

*   键值对集合，类似 Java 的 `HashMap<String, String>`
*   适合存储对象，可以单独操作对象的某个字段
*   节省内存（相比用 String 存储 JSON）

### 常用命令

```bash
HSET user:1001 name "张三" age 25
HGET user:1001 name
HGETALL user:1001
HINCRBY user:1001 age 1    # 年龄+1
```

```java
// 1. 存储用户信息
String key = "user:1001";
redisTemplate.opsForHash().put(key, "name", "张三");
redisTemplate.opsForHash().put(key, "age", 25);

// 2. 获取单个字段
String name = (String) redisTemplate.opsForHash().get(key, "name");
```

### 使用场景

| 场景           | 示例                      |
| :----------- | :---------------------- |
| **缓存对象型的数据** | 用户信息、商品信息、配置项           |
| **计数器集合**    | 多条统计性的数据 pv/uv/count 这些 |

## List（列表）

### 特性

*   有序、可重复的字符串集合
*   底层是双向链表，支持从两端压入/弹出
*   索引操作较慢（O(n)）

### 常用命令

```java
LPUSH queue:task task1 task2   # 左侧插入
RPUSH queue:task task3          # 右侧插入
LPOP queue:task                 # 左侧弹出
RPOP queue:task                 # 右侧弹出
LRANGE queue:task 0 -1          # 获取全部
```

### 使用场景

| 场景           | 示例                                                                              |
| :----------- | :------------------------------------------------------------------------------ |
| **消息队列**     | 生产者 RPUSH，消费者 LPOP（简单版） 不建议使用 建议直接采用rabbitmq kafka等 因为有ack机制  redis是ap模型消息可能会丢失 |
| **栈**        | LPUSH + LPOP（后进先出）                                                              |
| **队列**       | RPUSH + LPOP（先进先出）                                                              |
| **最新消息/时间线** | LPUSH + LTRIM 保持固定长度                                                            |
| **阻塞队列**     | BLPOP / BRPOP（阻塞式弹出） 也不建议使用                                                     |

## Set（集合）

### 特性

*   无序、不可重复的字符串集合 (如果都是整形的话是有序的)
*   支持交集、并集、差集运算
*   单元素操作 O(1)

### 常用命令

```java
SADD user:1001:tags "Java" "Redis" "Spring"   # 添加元素
SREM user:1001:tags "Java"                   # 移除元素
SMEMBERS user:1001:tags                      # 获取所有元素
SISMEMBER user:1001:tags "Redis"             # 判断是否存在
SINTER set1 set2                             # 交集
SUNION set1 set2                             # 并集
SDIFF set1 set2                              # 差集
```

### 使用场景

| 场景          | 示例                                           |
| :---------- | :------------------------------------------- |
| **标签系统**    | 用户标签、文章标签                                    |
| **共同好友**    | `SINTER user:1001:friends user:1002:friends` |
| **抽奖/去重**   | `SPOP lucky:draw 10`（随机抽取10人）                |
| **关注/粉丝关系** | 关注集合、粉丝集合                                    |
| **黑名单**     | `SADD blacklist:ip "192.168.1.1"`            |

## Sorted Set（有序集合）

### 特性

*   每个元素关联一个分数（score），按分数排序
*   元素不可重复，但分数可以重复
*   支持范围查询、排名查询

### 常用命令

```bash
ZADD rank:game 1000 player1 2000 player2   # 添加并设置分数
ZINCRBY rank:game 100 player1              # player1分数+100
ZRANGE rank:game 0 -1                      # 按分数升序
ZREVRANGE rank:game 0 -1                   # 按分数降序
ZRANK rank:game player1                    # 获取player1排名（0开始）
ZREVRANK rank:game player1                 # 倒序排名
ZREVRANGE rank:game 0 9                    # 获取前10名
```

### 使用场景

| 场景           | 示例               |    |
| :----------- | :--------------- | :- |
| **排行榜**      | 游戏积分榜、热销商品榜      |    |
| **延迟队列**     | 时间戳作为分数，定时轮询     |    |
| **范围查询**     | 地理位置范围（结合GEO更合适） |    |
| **带权重的任务队列** | 优先级作为分数          |    |
| **滑动窗口限流**   | 时间戳作为分数，统计窗口内请求数 |    |

## BitMap (位图)

### 特性

*   极致的内存效率 1亿个布尔值的内存占用\~12 MB
*   基于偏移量的 O(1) 随机访问 无论偏移量是 0 还是 1亿，时间复杂度都是 O(1)
*   自动动态扩容 注意：一次性的超大偏移量写入可能导致内存突然膨胀
*   高效的统计能力

### 常用命令

```bash
# 多个客户端同时执行这些操作，不会相互干扰
SETBIT key1 100 1  # 原子操作
GETBIT key1 100    # 原子操作
BITOP AND dest key1 key2  # 原子操作

# BITCOUNT：统计指定范围内1的数量
BITCOUNT sign:2025     # 统计全年签到次数

# BITPOS：查找第一个0或1的位置
BITPOS active:users 1  # 查找第一个在线用户
```

### 使用场景

| 场景              | 描述                 | Key 设计示例                                  | 价值           |
| :-------------- | :----------------- | :---------------------------------------- | :----------- |
| **用户签到**        | 记录用户每日签到，统计连续/累计签到 | `sign:userId:2025` (offset=天数-1)          | 亿级用户仅需几十MB内存 |
| **每日活跃用户(DAU)** | 统计当天登录用户           | `dau:2025-01-15` (offset=userId)          | 实时统计在线人数     |
| **用户留存分析**      | 计算次日/7日/30日留存率     | `active:2025-01-15` + `active:2025-01-16` | 位运算快速计算交集    |

### 布隆过滤器和布谷鸟过滤器的区别

####  支持删除操作 (最核心区别)

*   布隆过滤器：由于多个元素会共享同一个位，当你把某个位从1改为0来“删除”一个元素时，很可能会错误地删除其他元素的信息，因此它**不支持删除操作**。
*   布谷鸟过滤器：它存储的是元素的“指纹”（一段简短的哈希值），并且每个元素在桶里是独立存放的。因此，可以精确定位并删除某个元素的指纹，从而支持删除操作。需要注意的是，如果两个不同元素的指纹相同，删除时可能会发生“误删”。

根据上面的对比，你可以根据自己的需求来做选择：

*   首选布隆过滤器 (Bloom Filter) 的场景：

    *   你的数据是静态的，或者不需要删除。例如：防止缓存穿透、黑名单过滤。
    *   你能容忍一定的误判率，并希望实现简单、性能稳定。
    *   你的系统中有大量重复插入的可能性。
*   首选布谷鸟过滤器 (Cuckoo Filter) 的场景：

    *   你的应用明确需要支持删除操作。例如：需要维护一个动态变化的黑名单，过期的条目可以被移除。
    *   你对查询性能和内存占用有极致的要求，且可以接受其实现更复杂一些。
    *   你可以控制负载因子，确保数据量不会让过滤器接近满载。

> Redission实现了布隆过滤器

```java
import org.redisson.api.RBloomFilter;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Component;

@Component
public class BloomFilterService {

    private final RedissonClient redissonClient;

    // 构造器注入 RedissonClient
    public BloomFilterService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    /**
     * 获取或初始化布隆过滤器
     * @param name 过滤器名称（Redis中的Key）
     * @param expectedInsertions 预期插入数量
     * @param falseProbability 误判率（如：0.01 表示 1%）
     */
    public <T> RBloomFilter<T> getFilter(String name, long expectedInsertions, double falseProbability) {
        RBloomFilter<T> filter = redissonClient.getBloomFilter(name);
        // tryInit 仅在过滤器不存在时初始化，若已存在则不会覆盖配置
        filter.tryInit(expectedInsertions, falseProbability);
        return filter;
    }

    /**
     * 添加元素
     */
    public <T> boolean add(String filterName, T item) {
        RBloomFilter<T> filter = redissonClient.getBloomFilter(filterName);
        return filter.add(item);
    }

    /**
     * 检查元素是否存在
     * 返回 true 表示可能存在（有误判风险），返回 false 表示一定不存在
     */
    public <T> boolean mightContain(String filterName, T item) {
        RBloomFilter<T> filter = redissonClient.getBloomFilter(filterName);
        return filter.contains(item);
    }
}
```
