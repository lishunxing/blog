# 索引的作用

索引就是数据库管理系统里面的一个有序的数据结构, 用于加快查询和更新表中数据, InnoDB 使用的数据结构：B+树

将索引列和数据所在磁盘地址放在索引表中 就可以通过索引列的值直接查询到所在行地址 所以索引会这么快

![](https://file.lishunxing.cn/img/mysql%2F%E7%B4%A2%E5%BC%95%E7%BB%93%E6%9E%84.jpg)

## 聚簇索引 （Clustered Index）

聚簇索引也叫主键索引/聚集索引,  叶子节点存储：完整的行数据

特点：

*   一张表只能有一个（按主键组织数据）
*   如果没有主键，InnoDB 自动选择第一个唯一非空索引
*   如果也没有，自动生成隐藏的 6 字节 row\_id

![](https://file.lishunxing.cn/img/mysql%2F%E4%BA%8C%E5%8F%89%E6%A0%91.jpg)

## 二级索引

叶子节点存储：索引列的值 + 主键值

特点：

*   一张表可以有多个, Mysql默认最多64个
*   查询需要回表（除非索引覆盖）
*   手动去基于哪些字段建立索引时，会根据这些字段排序创建一个B+树，排序规则先根据第一个字段，第一个字段相等根据第二个，以此类推。
*   二级索引不会有完整的行数据，只有索引的字段以及主键索引的排序字段(主键ID)

# 索引的数据结构选择

在有序的数组下, 数据检索最快的方法是二分查找法 , 有序数组可以保证此数据要求, 但是对于数据的写入来讲 每次改变都会改变下标

所以需要使用链表来保存数据  基于有序数组+链表 所以就有了二叉树&#x20;

关于各种树的数据演示可以参考如下网址: <https://www.cs.usfca.edu/~galles/visualization/Algorithms.html>

| 数据结构   | 优点                             | 缺点                             |
| :--------- | :------------------------------- | :------------------------------- |
| 数组       | 随机访问快                       | 插入删除 O(n)，不适合动态数据    |
| 链表       | 插入删除快                       | 查找 O(n)                        |
| 哈希表     | 等值查询 O(1)                    | 范围查询慢，不支持排序           |
| 二叉树     | 查找 O(log n)                    | 可能退化为链表                   |
| AVL/红黑树 | 平衡，O(log n)                   | 树太高，磁盘 I/O 多              |
| B树        | 矮胖，磁盘友好                   | 非叶子节点也存数据，空间利用率低 |
| **B+树**   | 更矮胖，范围查询快，空间利用率高 | -                                |

## 二叉树

二叉树在特定情况下 会是一颗斜树 会转换为链表形式  比如写入1 2 3 4 5 6 7 就会变成一个单链表 不够平衡 所以扫描全表  基于此情况 衍生出了平衡二叉树 (AVL树)

## 平衡二叉树 (AVL树)

平衡二叉树的左右子树深度绝对值不能超过1

![](https://file.lishunxing.cn/img/mysql%2F%E5%B9%B3%E8%A1%A1%E4%BA%8C%E5%8F%89%E6%A0%91-%E7%AE%80%E8%A6%81.jpg)

如果写入了123456 那么左子树深度是2  右边的深度就不能超过2  左子树是比有子树数量多 基于左旋和右旋  来保持平衡

![](https://file.lishunxing.cn/img/mysql%2F%E5%B9%B3%E8%A1%A1%E4%BA%8C%E5%8F%89%E6%A0%91-%E8%AF%A6%E7%BB%86.jpg)

这样来保存数据的话搜索速度是可以保证了  但是存在性能问题

1.  就是树的节点是保存在磁盘中的   innodb的数据页是16KB  这样保存数据是远远达不到16KB的大小 会造成存储空间的浪费
2.  每一次查找找树的节点, 都会进行下一块磁盘内容的加载 , 性能会有所影响 所以为了解决以上的问题 引出了多路平衡查找树 B树

为了解决上述问题, 引入了B树

## 多路平衡查找树(B树)

![](https://file.lishunxing.cn/img/mysql%2F%E5%A4%9A%E8%B7%AF%E5%B9%B3%E8%A1%A1%E6%9F%A5%E6%89%BE%E6%A0%91\(B%E6%A0%91\).jpg)

1.  在每个节点保存多个数值, 大大的减少了树的深度. 可以根据空间来指定键值数量 树的分叉是关键字数量+1 通过分裂来保持树的平衡 &#x20;
2.  此时数据已经可以解决搜索和性能问题了 但是为了保持扫库扫表的性能 MYSQL使用了B+树来保存数据

## 增强型多路平衡查找树(B+树)

![](https://file.lishunxing.cn/img/mysql%2F%E5%A2%9E%E5%BC%BA%E5%9E%8B%E5%A4%9A%E8%B7%AF%E5%B9%B3%E8%A1%A1%E6%9F%A5%E6%89%BE%E6%A0%91\(B%2B%E6%A0%91\).jpg)

1.  B+树数据存储结构如上图所所示 所有的数据节点都在叶子节点保存  普通节点只保存索引信息&#x20;
2.  叶子节点会形成一个有序链表 比如如果查找 age > 20 and age < 60 直接找到20的开始值  然后查找后面的地址即可

## B+树相对于B树的优势

1.  B树能解决的问题, B+树都能解决
2.  扫库, 扫表能力更强
3.  磁盘读写能力更强
4.  排序能力更强
5.  效率更加稳定

# 索引的建立使用原则&#x20;

## 应该建索引的场景

1.  用于在Where判断, Order排序, JOIN的字段, Group By 的字段创建
2.  离散度高的字段
3.  尽量选择较小的数据类型: 索引列越小，B+树每页存储的索引项越多 → 树越矮 → I/O 越少。
4.  覆盖索引

## 不应该建索引的场景

1.  离散度极低: 性别（男女比例1:1）、状态（只有0/1且分布均匀）, 过滤效果差，甚至不如全表扫描
2.  频繁更新的列: 登录时间、最后活跃时间, 索引维护成本高
3.  长字段: TEXT、BLOB、超长 VARCHAR, 占用空间大，B+树扇出低
4.  很少使用的列: 浪费空间和维护成本
5.  小表: 配置表、字典表, 全表扫描更快（几百行）

## 列的离散度&#x20;

当离散度越高的情况下  就是比例越小的情况下 才去建立索引 如果是性别这种 只有男女2个值  离散度特别低  建立索引是无效的&#x20;

```sql
-- 区分度 = 唯一值数量 / 总行数
SELECT COUNT(DISTINCT column) / COUNT(*) FROM table;
```

| 区分度      | 评价 | 是否建索引       |
| :---------- | :--- | :--------------- |
| > 0.1       | 较高 | ✅ 效果好         |
| 0.01 \~ 0.1 | 一般 | ⚠️ 谨慎评估       |
| < 0.01      | 很低 | ❌ 效果差，不推荐 |

## 复合索引（联合索引）设计原则

### 最左前缀原则

复合索引遵循从左到右的使用顺序：

```sql
-- 索引：(name, age, city)

✅ 能使用索引：
WHERE name = '张三'                     -- 使用 name
WHERE name = '张三' AND age = 25        -- 使用 name, age
WHERE name = '张三' AND age = 25 AND city = '北京'  -- 全部使用

❌ 不能使用索引：
WHERE age = 25                          -- 没有 name
WHERE city = '北京'                     -- 没有 name
```

### 区分度高的列放在左边

```sql
-- 假设：user_id 区分度极高（每个值唯一），status 区分度低（只有 0/1）

-- ✅ 推荐：高区分度在左
CREATE INDEX idx_user_status ON orders(user_id, status);

-- ❌ 不推荐：低区分度在左
CREATE INDEX idx_status_user ON orders(status, user_id);
```

原因：高区分度列在左边可以快速缩小查询范围。

### 等值查询的列放在左边，范围查询的列放在右边

```sql
-- 查询模式：WHERE user_id = ? AND create_time BETWEEN ? AND ?

-- ✅ 推荐：等值条件在左，范围条件在右
CREATE INDEX idx_user_time ON orders(user_id, create_time);

-- ❌ 不推荐：范围条件在左
CREATE INDEX idx_time_user ON orders(create_time, user_id);
```

原因：范围查询（>、<、BETWEEN、LIKE）会让右边的列无法使用索引。

> 如果根据索引查出来的数据是无序的, 那么就无法使用索引。

## 覆盖索引设计原则

定义：索引包含查询所需的所有字段，无需回表。

```sql
-- 查询需要 name 和 age
SELECT name, age FROM users WHERE name = '张三';

-- ✅ 覆盖索引：索引包含 name 和 age，无需回表
CREATE INDEX idx_name_age ON users(name, age);

-- ❌ 非覆盖索引：索引只有 name，需要回表读取 age
CREATE INDEX idx_name ON users(name);
```

何时优先考虑覆盖索引：

*   高频查询的字段集
*   需要减少随机 I/O 的场景
*   二级索引查询频繁的场景

## 索引下推（Index Condition Pushdown，ICP）

索引下推是 MySQL 5.6 引入的一项查询优化技术，它允许 MySQL 在存储引擎层就利用索引过滤掉不符合条件的记录，从而减少回表次数。

一句话定义：将 WHERE 条件中能用索引过滤的部分，下推到存储引擎层提前执行，减少回表到 Server 层的次数。只有存在复合索引的情况下, 才会有索引下推。

> ICP 的"下推"指的是将索引中已有的列的条件提前过滤

### 为什么要索引下推

假设有一个复合索引 (name, age)：

```sql
CREATE INDEX idx_name_age ON users(name, age);
```

执行查询：

```sql
SELECT * FROM users 
WHERE name LIKE '张%' 
  AND age = 25;
```

没有 ICP（MySQL 5.6 之前）

```text
步骤：
1. 使用索引找到所有 name LIKE '张%' 的记录（可能有 1000 条）
2. 对这 1000 条记录逐条回表（1000 次随机 I/O）
3. 回表后，在 Server 层过滤 age = 25（可能只剩 10 条）

问题：回表了 1000 次，实际只需要 10 条
```

有 ICP（MySQL 5.6+）

```text
步骤：
1. 使用索引找到所有 name LIKE '张%' 的记录（1000 条）
2. 在存储引擎层，直接检查这 1000 条记录的 age 字段（age 也在索引中）
3. 只对 age = 25 的记录回表（10 次随机 I/O）

优势：回表次数从 1000 次降到 10 次
```

### 不能下推的条件类型

*   子查询
*   存储过程/函数（非确定性函数）
*   某些类型的类型转换

### 如何确认 ICP 是否生效

查看执行计划

```sql
EXPLAIN SELECT * FROM users WHERE name LIKE '张%' AND age = 25;
```

`Extra` 列显示 `Using index condition` → ICP 已生效。

### 控制 ICP 开关

```sql
-- 查看 ICP 状态
SHOW VARIABLES LIKE 'optimizer_switch';
-- 输出中的 index_condition_pushdown=on/off

-- 开启 ICP
SET optimizer_switch = 'index_condition_pushdown=on';

-- 关闭 ICP
SET optimizer_switch = 'index_condition_pushdown=off';
```

## 索引什么时候会失效

1.  在索引列上使用函数, 或者表达式 比如+1 操作
2.  字符串没有带引号, 出现隐式转换 例如 `WHERE phone = 13800000000`（phone 是 VARCHAR）, 应该改为 `WHERE phone = '13800000000'`
3.  like条件中最左边带%
4.  反向查询  比如like可以匹配索引 not like有时候不匹配  这种情况不是绝对的 是基于优化器来进行查询的 默认就是基于成本的优化器去进行选择
5.  复合索引不满足最左前缀
6.  OR 条件
7.  使用 != 或 <>

# 索引优化

## 检查现有索引的效率

### 查看索引使用情况

```sql
-- MySQL 5.6+，查看索引是否被使用
SELECT * FROM sys.schema_unused_indexes;

-- 查看未使用的索引（需要开启 performance_schema）
SELECT * FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL AND count_star = 0;
```

### 查找重复索引

```sql
-- 查看表的索引信息
SHOW INDEX FROM your_table;
```

常见重复情况：

*   `(name)` 和 `(name, age)`：`(name)` 是冗余的
*   `(name, age)` 和 `(age, name)`：不同用途，可能都需要

### 使用 EXPLAIN 验证执行计划

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM users WHERE name = '张三';

-- 查看更详细的信息（MySQL 8.0+）
EXPLAIN FORMAT=TREE SELECT * FROM users WHERE name = '张三';

-- 查看实际执行时的统计信息（MySQL 8.0.18+）
EXPLAIN ANALYZE SELECT * FROM users WHERE name = '张三';
```

关键列：

*   `type`：`ref`/`range` 比 `ALL` 好
*   `key`：是否使用了预期的索引
*   `rows`：扫描行数
*   `Extra`：`Using index`（覆盖索引）、`Using filesort`（需要优化）

| 字段            | 说明                             |      |
| :-------------- | :------------------------------- | :--- |
| `id`            | 查询中每个 SELECT 子句的标识符   |      |
| `select_type`   | SELECT 的类型                    |      |
| `table`         | 当前行正在访问的表名             |      |
| `partitions`    | 匹配的分区（未分区则为 NULL）    |      |
| `type`          | **连接类型**（最重要的字段之一） |      |
| `possible_keys` | 可能使用的索引                   |      |
| `key`           | **实际使用的索引**               |      |
| `key_len`       | 使用的索引长度（字节）           |      |
| `ref`           | 与索引比较的列或常量             |      |
| `rows`          | **预计扫描的行数**               |      |
| `filtered`      | 经过表条件过滤后剩余行的百分比   |      |
| `Extra`         | **额外信息**（含金量高）         |      |

#### type：连接类型

按性能从高到低排序：

| type              | 含义        | 说明                                       |
| :---------------- | :---------- | :----------------------------------------- |
| `system`          | 系统表      | 只有一行数据，特例                         |
| `const`           | 常量        | 主键或唯一索引等值查询，最多返回一行       |
| `eq_ref`          | 等值引用    | 连接查询中，使用主键或唯一索引，只匹配一行 |
| `ref`             | 引用        | 非唯一索引等值查询，可能返回多行           |
| `fulltext`        | 全文索引    | 使用 FULLTEXT 索引                         |
| `ref_or_null`     | 引用或 NULL | 类似 ref，但额外查找 NULL 值               |
| `index_merge`     | 索引合并    | 使用多个索引合并结果                       |
| `unique_subquery` | 唯一子查询  | IN 子查询返回唯一值                        |
| `index_subquery`  | 索引子查询  | IN 子查询返回索引值                        |
| `range`           | 范围扫描    | 使用索引进行范围查询（>、<、BETWEEN、IN）  |
| `index`           | 全索引扫描  | 扫描整个索引树（比 ALL 快）                |
| `ALL`             | 全表扫描    | **最差**，扫描所有行                       |

**优化目标**：至少达到 `range` 级别，争取达到 `ref` 或 `const`。

```sql
-- const：主键等值查询
EXPLAIN SELECT * FROM users WHERE id = 1;

-- ref：普通索引等值查询
EXPLAIN SELECT * FROM users WHERE name = '张三';

-- range：范围查询
EXPLAIN SELECT * FROM users WHERE age BETWEEN 18 AND 25;

-- ALL：全表扫描（危险）
EXPLAIN SELECT * FROM users WHERE age = 18;  -- 如果 age 没有索引
```

#### possible\_keys(优化器认为可能使用的索引列表) 和 key(优化器实际选择的索引)

*   `key = NULL`：没有使用索引（通常是 `type = ALL`）
*   `possible_keys` 有值但 `key` 为 NULL：优化器评估后认为索引不划算

```sql
-- 强制使用索引
SELECT * FROM users FORCE INDEX(idx_name) WHERE name = '张三';
```

#### rows：预估扫描行数

优化器预估的需要读取的行数。值越小越好。

```ql
-- 预估扫描 1000 行
EXPLAIN SELECT * FROM users WHERE age = 18;  -- age 无索引
-- rows ≈ 表的总行数

-- 预估扫描 10 行
EXPLAIN SELECT * FROM users WHERE id = 1;    -- 主键索引
-- rows = 1
```

> ⚠️ rows 是预估值，不是精确值。`EXPLAIN ANALYZE` 可以看到实际行数。

#### Extra：额外信息（⭐ 含金量高）

| Extra 值                       | 含义                                    | 优化建议           |
| :----------------------------- | :-------------------------------------- | :----------------- |
| `Using index`                  | **覆盖索引**，无需回表                  | ✅ 最优             |
| `Using where`                  | 使用了 WHERE 过滤                       | 正常               |
| `Using index condition`        | 使用**索引下推（ICP）**                 | ✅ 好               |
| `Using filesort`               | 需要额外排序（文件排序）                | ❌ 需要优化，加索引 |
| `Using temporary`              | 使用临时表（常见于 GROUP BY、DISTINCT） | ❌ 需要优化         |
| `Using join buffer`            | 连接缓冲区（没有使用索引连接）          | ❌ 需要加索引       |
| `Using MRR`                    | 使用多范围读取优化                      | ✅ 好               |
| `Impossible WHERE`             | WHERE 条件永远为假                      | 检查 SQL           |
| `No tables used`               | 没有 FROM 或 `FROM DUAL`                | 正常               |
| `Select tables optimized away` | 聚合函数直接返回值                      | ✅ 最优             |
| `Distinct`                     | 优化了 DISTINCT                         | ✅ 好               |
| `Using union(...)`             | 索引合并                                | 正常               |
| `Using intersect(...)`         | 索引交集                                | 正常               |
| `LooseScan`                    | 松散的索引扫描（GROUP BY 优化）         | ✅ 好               |

#### Using filesort（需要优化）：

```sql
-- 没有索引的 ORDER BY
EXPLAIN SELECT * FROM users ORDER BY create_time;
-- Extra: Using filesort

-- 加索引后
CREATE INDEX idx_create_time ON users(create_time);
-- Extra: (空) 或 Using index

-- GROUP BY 顺序与索引不一致
EXPLAIN SELECT name, COUNT(*) FROM users GROUP BY age, name;
-- 索引是 (name, age) 时可能产生临时表

-- 调整 GROUP BY 顺序与索引一致可优化
```

## COUNT、LIMIT、排序、分组优化

### COUNT优化

COUNT(\*) 或者 COUNT(1) 都可以， 不要COUNT(字段）因为 COUNT(字段名) 会忽略该字段为 NULL 的行，而 COUNT(\*) 和 COUNT(1) 不会。

### LIMIT 优化

LIMIT 用于限制查询返回的行数，但随着偏移量（OFFSET）增大，查询会越来越慢。这是 LIMIT 最常见的性能问题。

```sql
-- 前几页很快
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 0;     -- 快
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 100;   -- 还行

-- 后面几页非常慢
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 100000; -- 慢！
```

#### 先查主键，再关联原表

```sql
SELECT u.* 
FROM users u
INNER JOIN (
    SELECT id FROM users ORDER BY id LIMIT 100000, 10
) AS tmp ON u.id = tmp.id;
```

原理：

*   子查询只扫描主键索引（覆盖索引），不读取完整行
*   外层查询只取需要的 10 行数据

#### 使用游标分页（记住上次位置）

```sql
-- 第 1 页：记录最后一条的 id = 10
SELECT * FROM users ORDER BY id LIMIT 10;

-- 第 2 页：直接从上一次的 id 开始
SELECT * FROM users WHERE id > 10 ORDER BY id LIMIT 10;

-- 第 N 页：WHERE id > 上一次的最大 id
SELECT * FROM users WHERE id > 上一次的id ORDER BY id LIMIT 10;
```

优点：

*   无论翻到第几页，性能都恒定（O(log n)）
*   不需要扫描跳过的行

缺点：

*   不支持跳页（只能上一页/下一页）
*   需要记住上一页的最后位置

#### 限制用户最大翻页数

业务层面限制：最多只能看前 1000 条

### ORDER BY / GROUP BY 优化

ORDER BY 用于对查询结果排序，但如果处理不当，会导致 Using filesort（文件排序），产生临时文件和磁盘 I/O，严重影响性能。

GROUP BY 用于对数据进行分组聚合，但如果处理不当，会导致 Using temporary（使用临时表）和 Using filesort（文件排序），严重影响性能。

*   添加合适的索引
*   复合索引优化

## 慢查询优化

### 开启慢查询日志

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志（MySQL 8.0+）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录
SET GLOBAL log_queries_not_using_indexes = ON;  -- 记录未使用索引的查询

-- 永久配置（my.cnf）
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1
```

### 分析慢查询日志

```bash
# 使用 mysqldumpslow 分析
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# -s t：按查询时间排序
# -t 10：显示前 10 条
```

### 实时查看当前慢查询

```sql
-- 查看当前正在执行的查询
SHOW PROCESSLIST;

-- 查看性能模式中的慢查询
SELECT * FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC LIMIT 10;
```

### 慢查询的常见原因

| 原因           | 说明                           | 占比 |
| :------------- | :----------------------------- | :--- |
| **索引缺失**   | WHERE/ORDER BY/JOIN 列没有索引 | 40%  |
| **索引失效**   | 函数、隐式转换、LIKE '%xx'     | 20%  |
| **查询设计差** | SELECT \*、N+1 查询、重复查询  | 15%  |
| **数据量过大** | 未做分页、历史数据未清理       | 10%  |
| **锁等待**     | 行锁/表锁竞争                  | 10%  |
| **配置不当**   | Buffer Pool 太小、连接数不足   | 5%   |

> 建议通过EXPLAIN字段, 分析慢SQL进行优化

1.  表结构是否合理，包括是否有合适的字段类型、合适的行格式、合适的存储引擎;是否有针对线上不同的数据量进行库、表的横向、纵向拆分。&#x20;
2.  是否建立合适的索引，索引是否生效，是否有用到覆盖索引，是否有用到优化器的相关优化，比如count、group by、order by等等。&#x20;
3.  是否基于场景设置合理的内存缓存区间，比如InnoDB缓冲池和MylISAM键缓存。&#x20;
4.  是否用到合理的锁定策略。比如InnoDB里面不同的隔离级别会性能与一致性之间有取舍。
