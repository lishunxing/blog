# 循环依赖与三级缓存

## 简介

循环依赖是指多个 Bean 之间相互引用，形成闭环。Spring 通过三级缓存机制解决了大部分循环依赖问题。

### 什么是循环依赖

```java
// 1. 构造器循环依赖（无法解决）
@Component
public class A {
    public A(B b) { }  // A 依赖 B
}

@Component
public class B {
    public B(A a) { }  // B 依赖 A
}

// 2. Setter/字段循环依赖（Spring 能解决）
@Component
public class A {
    @Autowired private B b;  // A 依赖 B
}

@Component
public class B {
    @Autowired private A a;  // B 依赖 A
}
```

---

## 三级缓存机制

Spring 解决循环依赖的核心是三级缓存（实际是三个 Map）。

| 缓存 | 变量名 | 存储内容 | 作用 |
| :--- | :--- | :--- | :--- |
| **一级缓存** | singletonObjects | 完全初始化好的 Bean | 已完成实例化、属性注入和初始化，可供应用程序直接使用 |
| **二级缓存** | earlySingletonObjects | 提前暴露的原始 Bean（半成品） | 已实例化但尚未进行属性注入和初始化，用于提前暴露解决循环依赖 |
| **三级缓存** | singletonFactories | ObjectFactory（工厂对象） | 存储 ObjectFactory 接口的实现，支持 AOP 等高级功能 |

```java
/** 一级缓存：完全体 Bean */
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** 二级缓存：提前暴露的原始 Bean（半成品） */
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

/** 三级缓存：对象工厂 */
private final Map<String, ObjectFactory<?>> singletonFactories = new ConcurrentHashMap<>(16);
```

## 源码分析

以"UserService 依赖 OrderService，OrderService 依赖 UserService "的场景为例。

### 第一步：Bean 创建准备（进入三级缓存）

```java
// AbstractAutowireCapableBeanFactory.doCreateBean() 假设这里是对UserService进行创建
protected Object doCreateBean(String beanName, RootBeanDefinition mbd, @Nullable Object[] args) {
    // 1. 实例化 Bean（调用构造函数）
    BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);
    Object bean = instanceWrapper.getWrappedInstance();
    
    // 2. 判断是否允许提前暴露（解决循环依赖）
    boolean earlySingletonExposure = (mbd.isSingleton() && 
                                       this.allowCircularReferences && 
                                       isSingletonCurrentlyInCreation(beanName));
    
    if (earlySingletonExposure) {
        // 3. 将 ObjectFactory 放入三级缓存 这里放入了UserService
        addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));
        // 此时：三级缓存有工厂，二级缓存无，一级缓存无
    }
    
    // 4. 填充属性（可能触发循环依赖） 在这里依赖注入的时候UserService会依赖OrderService
    populateBean(beanName, mbd, instanceWrapper);
    
    // 5. 初始化 Bean（如调用 init-method，这里可能会产生 AOP 代理对象）
    exposedObject = initializeBean(beanName, bean, mbd);
    
    return exposedObject;
}
```

**此时状态**：

- **三级缓存**：`singletonFactories["userService"] = ObjectFactory`
- **二级缓存**：`earlySingletonObjects["userService"] = null`
- **一级缓存**：`singletonObjects["userService"] = null`

### 第二步：循环依赖触发（三级→二级转换）

```java
// 在 populateBean 中注入依赖
protected void populateBean(String beanName, ...) {
    // 假设 userService 依赖 orderService
    // 调用 getBean("orderService")
}

// AbstractBeanFactory.doGetBean()
protected <T> T doGetBean(String name, ...) {
    // 尝试从缓存获取
    Object sharedInstance = getSingleton(beanName);
    // 进入 getSingleton 方法
}

// DefaultSingletonBeanRegistry.getSingleton()
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 1. 从一级缓存获取
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        // 2. 从二级缓存获取
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            // 3. 从三级缓存获取 ObjectFactory
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
                // 4. 调用工厂获取提前引用
                singletonObject = singletonFactory.getObject();
                // 5. 将提前引用放入二级缓存（三级→二级转换）
                this.earlySingletonObjects.put(beanName, singletonObject);
                // 6. 从三级缓存移除
                this.singletonFactories.remove(beanName);
            }
        }
    }
    return singletonObject;
}
```

**此时状态（UserService 被提前暴露给 OrderService）**：

- **三级缓存**：`singletonFactories["userService"] = null`（已移除）
- **二级缓存**：`earlySingletonObjects["userService"] = 半成品UserService`
- **一级缓存**：`singletonObjects["userService"] = null`

### 第三步：Bean 完成初始化（二级→一级转换）

```java
// DefaultSingletonBeanRegistry
protected void addSingleton(String beanName, Object singletonObject) {
    // 从一级缓存获取写锁
    synchronized (this.singletonObjects) {
        // 1. 放入一级缓存
        this.singletonObjects.put(beanName, singletonObject);
        // 2. 从二级缓存移除
        this.earlySingletonObjects.remove(beanName);
        // 3. 从三级缓存移除
        this.singletonFactories.remove(beanName);
        // 4. 更新已注册单例集合
        this.registeredSingletons.add(beanName);
    }
}
```

**最终状态（完全初始化后）**：

- **三级缓存**：`null`
- **二级缓存**：`null`
- **一级缓存**：`singletonObjects["userService"] = 完整UserService`

## 无法解决的循环依赖类型

| 类型 | 说明 |
| :--- | :--- |
| **构造器注入循环依赖** | 因为实例化本身就需要依赖对方，导致对象都无法被创建出来，无法提前暴露 |
| **prototype 作用域的 Bean** | Spring 不缓存原型 Bean，因此无法处理 |
| **特定 AOP 代理（如 @Async）复杂场景** | 当一个代理对象的创建时机与 Spring 的生命周期有冲突时，也可能导致循环依赖异常 |

---

## 为什么需要三级缓存

纯属性注入理论上二级缓存就够了，三级缓存的核心目的是为了支持 AOP，并保持代理创建时机的优雅。

### 二级缓存方案的核心困境

| 缓存名 | 存储内容 |
| :--- | :--- |
| 一级缓存 (singletonObjects) | 成品 Bean |
| 二级缓存 (earlySingletonObjects) | 半成品 Bean（可被提前暴露） |

#### 问题：放入二级缓存时应该放什么？

| 选项 | 问题 |
| :--- | :--- |
| 放原始对象 | 如果 A 最终需要被 AOP 代理，B 拿到的是原始对象，AOP 失效 |
| 放最终对象（代理） | 需要在实例化后立即决定最终形态，违反 Spring 的设计原则 |

### Spring 的设计原则

> 核心逻辑：让 Bean 的生命周期保持纯粹与完整，将 AOP 代理视为对原始 Bean 的一种"增强装饰"，而非其固有属性。

| 阶段 | 说明 |
| :--- | :--- |
| 实例化 (createBeanInstance) | 得到一个原始对象 |
| 属性填充 (populateBean) | 注入依赖 |
| 初始化 (initializeBean) | 执行 Aware、BeanPostProcessor、@PostConstruct、init-method，后置处理时生成代理 |

### 三级缓存如何优雅解决

```java
// 三级缓存存的不是对象，而是工厂
addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));
```

| 场景 | 二级缓存方案 | 三级缓存方案 |
| :--- | :--- | :--- |
| 普通 Bean（无循环依赖） | 实例化后立即检查 AOP → 浪费性能 | 只存工厂 → 不检查 AOP → 初始化时才检查 |
| 有 AOP 且无循环依赖 | 实例化后立即创建代理 → 时机过早 | 初始化时才创建代理 → 时机正确 |
| 有 AOP 且有循环依赖 | 实例化后立即创建代理 → 可以工作 | 被依赖时在 getEarlyBeanReference 中创建代理 → 也可以工作 |

## getEarlyBeanReference 详解

getEarlyBeanReference 是 Spring 解决 AOP + 循环依赖问题的核心钩子方法。

### 源码分析

```java
// AbstractAutowireCapableBeanFactory
protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
    Object exposedObject = bean;
    // 遍历所有 SmartInstantiationAwareBeanPostProcessor
    for (BeanPostProcessor bp : getBeanPostProcessors()) {
        if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            SmartInstantiationAwareBeanPostProcessor ibp = (SmartInstantiationAwareBeanPostProcessor) bp;
            exposedObject = ibp.getEarlyBeanReference(exposedObject, beanName);
        }
    }
    return exposedObject;
}
```

### AbstractAutoProxyCreator 的实现

```java
// AbstractAutoProxyCreator
public Object getEarlyBeanReference(Object bean, String beanName) {
    // 缓存当前 Bean 的原始引用，用于后续比较
    this.earlyProxyReferences.put(beanName, bean);
    // 核心：创建代理对象（如果需要）
    return wrapIfNecessary(bean, beanName);
}
```

> 如果 Bean 不需要 AOP，wrapIfNecessary 直接返回原始对象。

## 构造器注入为何无法解决

解决循环依赖的前提是：Bean 已经实例化完成，有了可用的对象引用，才能将其提前暴露出去。

| 注入方式 | 流程 | 结果 |
| :--- | :--- | :--- |
| 属性注入 | 实例化 → 暴露工厂 → 填充属性 | 对象已存在，可以被其他 Bean 引用 |
| 构造器注入 | 实例化 → 需要依赖 → 无法暴露 | 形成死锁，根本没有机会把 A 暴露出去 |

---

## @Async 与 @Transactional 循环依赖差异

| 注解 | 代理类型 | getEarlyBeanReference 表现 | 循环依赖支持 |
| :--- | :--- | :--- | :--- |
| **@Transactional** | CGLIB 动态代理 | 能识别并创建代理 | ✅ 正常 |
| **@Async** | AsyncAnnotationBeanPostProcessor | 默认不实现 SmartInstantiationAwareBeanPostProcessor | ❌ 报错 |

### 原因

@Async 使用 AsyncAnnotationBeanPostProcessor，它在 getEarlyBeanReference 阶段默认不会为当前 Bean 创建代理，导致容器中存在两个不同的"A"：B 持有原始对象，A 完成初始化后产生代理对象，从而引发 BeanCurrentlyInCreationException 异常。

### 解决方案

1. 重新设计依赖关系，尽量避免在带有 @Async 的 Bean 上产生循环依赖
2. 调整注入方式：将属性注入改为构造器注入或 setter 注入
3. 使用 @Lazy：在注入点添加 @Lazy 注解，为依赖项生成一个代理，可以打破循环
